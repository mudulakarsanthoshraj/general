Summary

One design system, built once, used natively by every framework — approached as a measured, phased path, not a big-bang rewrite.

In one minute

  • What we have today: a shared Luma theme on top of Kendo UI — one CSS build, one npm package, used by React, Angular, Web Components, jQuery, Vue, and .NET. That aligns branding across stacks, but components are still Kendo’s — we cannot change their DOM structure or redesign them beyond what theming and vendor APIs allow.
  • The problem: each framework still depends on its own Kendo (or related vendor) build and documentation path, so stacks drift and work is repeated. Deeper UX, accessibility, or behaviour changes often hit those vendor limits instead of shipping once from our own components.
  • The fix: one shared core — design tokens, then web components, then thin framework wrappers — documented in a single Storybook.
  • The payoff: a fix is made once and ships to every framework, instead of being repeated three times.

What we are asking for today

  • Approve Phase 1 + Phase 2 only: publish design tokens, then run a short Stencil-vs-Lit spike on one component (Button) and write down the tool decision.
  • Cost: a small slice of existing team time — no new headcount, no licences.
  • Not in this ask: the full build. Phases 3–5 need separate approval once each gate shows results.

Why We Need This

We already reach multiple frameworks through a Kendo-based theme — but that is not the same as owning the components. Without a single component source we cannot freely change structure or design, and cost grows with every stack.

Today the Luma design system is primarily a theme layer on Kendo UI, used from React, Angular, Web Components, jQuery, Vue, and .NET / Blazor. That spreads Luma branding, but Kendo still owns each component’s markup, behaviour, and API. We can skin and configure — we cannot redesign DOM structure or ship custom component implementations the way a true multi-framework design system can.

  • Structural or interaction changes (layout, accessibility fixes, new variants) are limited by what Kendo exposes — not by what product or design need.
  • Kendo is per framework (React, Angular, jQuery, etc.) — different vendor packages and docs paths, even though the Luma theme CSS is the same everywhere.
  • Consuming teams still see drift between stacks — Kendo markup and behaviour differ by framework, undermining a truly identical experience.
  • Onboarding a new framework means wiring another Kendo (or vendor) stack, not installing one shared component package.

The trigger
Theming a vendor library got us a shared look, but it caps how far we can take our own design. With two or more active consumer frameworks and a real need to change structure and behaviour, industry practice is one web-component core we own plus thin wrappers — not a theme re-applied on someone else’s components per stack.

What this gets us — and where it stops

What the theme layer gives us

  • One theme, one npm package — a single CSS file serves React, Angular, jQuery, Vue, and .NET.
  • Fast reach — Kendo already ships components for each stack.
  • Mature, supported widgets (grids, pickers) without building them.
  • Familiar APIs for teams already using Kendo.

Where it stops

  • No control of DOM structure — markup is Kendo’s; we can’t restructure components.
  • Limited redesign — we style within what Kendo’s CSS hooks allow, not a blank canvas.
  • Behaviour is fixed — interaction and accessibility depend on the vendor’s implementation.
  • Kendo is still per framework — different vendor packages per stack; only the theme CSS is unified.
  • Vendor lock-in & licensing — tied to Kendo versions, upgrades, and licence terms.
Why this matters for the proposal
The theme-on-Kendo model is a sensible starting point, not a dead end. The proposed approach keeps the layer we already own (tokens) and gradually replaces the vendor-owned layers with our own web components — so we gain control of structure and design without losing cross-framework reach. See Proposed Approach.

Option Summary Why it falls short
A. Status quo Keep Kendo + per-framework Storybook paths as today. Lowest short-term effort; drift and duplicate work grow with adoption. Each new framework multiplies maintenance.
B. Separate design system per framework Dedicated React DS, Angular DS, Vue DS teams. ~3× component work, ~3× accessibility audits. Hard to keep visual parity; expensive at scale.
C. Buy a commercial design system License a vendor SaaS / component library. Often $10K–$50K+ per year, generic styling, weak fit for Luma-specific Kendo + Tabler patterns.

Cost of Inaction

What we pay if we keep the current multi-path model.

  • Duplicate engineering: Every component change (variant, a11y fix, token rename) is repeated across framework surfaces — roughly proportional to the number of active stacks.
  • Inconsistent UX: Teams and auditors notice when Angular and React apps don’t match; rework lands on the design system team anyway.
  • Slower adoption: New product lines on Vue or .NET wait for “their” Luma design system path instead of installing a published package.
  • Documentation debt: Storybook and Slack support scale with questions × frameworks, not with a single source of truth.
Directional, not precise
We are not claiming a single dollar ROI figure without baseline data. Phase 1 gives measurable token adoption; Phase 3 gives before/after maintenance time on three pilot components.

Ways to Build a Design System

Every design system enforces the same brand and design guidelines — the difference is the architecture underneath. Each option trades speed and effort against how much of the markup, behaviour, and design you actually control. Here are the seven realistic ways, what each gives you, and where it stops.

Faster / less control More control / more effort

Roughly, the options below run from least control (theme a vendor library) to most control (own components for every framework). Colour key in each diagram: orange = we own, red dashed = vendor owns.

1 Theme on a vendor component library Luma Design System (Theme) Control: Low–Med Speed: Fast

Apply brand tokens and build one Luma theme (a single CSS bundle) on top of Kendo’s component classes, published as an npm package and imported by apps in any framework. Kendo still ships a different component package per stack (React, Angular, jQuery, etc.) — but the theme itself is built once and served everywhere. This is how Luma works today.

Design tokens → One Luma theme (single CSS · npm) → Any framework app

Underneath: Kendo components (vendor-owned, different package per framework) provide the markup and behaviour the theme styles.

Strengths
  • One theme build, one npm package — same CSS in React, Angular, jQuery, Vue, and .NET apps.
  • Fast reach — Kendo already ships widgets per framework.
  • Mature grids, pickers, etc. without building them.
  • Low cost to maintain the theme layer.
Limits
  • No control of DOM structure or behaviour (Kendo owns that).
  • Redesign capped by what Kendo’s CSS hooks allow.
  • Still tied to Kendo per framework for components; vendor lock-in.

Best when: speed matters more than bespoke design, the team is small, or only one or two frameworks are in play.

2 Single-framework component library NOVUS Design System Control: High Speed: Medium

Build your own components in one framework (e.g. a React-only library) on top of shared design tokens. You own markup, behaviour, and API — but only for that stack.

We have already done this: the NOVUS Design System was built this way — our own components in a single framework, full control of design and implementation, published as a package for product teams on that stack. It proves the model works when one framework is enough; Luma’s challenge is reaching all frameworks without repeating that build three times.

Design tokens → Own components (one framework) → Apps on that stack

NOVUS Design System follows this pattern in production today.

Strengths
  • Full design and behaviour control in that framework.
  • Idiomatic, best-in-class developer experience.
  • Simplest build when there is only one framework.
  • Proven internally via NOVUS Design System.
Limits
  • Only serves one framework (NOVUS does not cover Angular, jQuery, .NET, etc.).
  • Other stacks get nothing or must re-implement from scratch.
  • Painful the day a second framework appears — why Luma looks beyond this.

Best when: essentially all consumers are one framework for the foreseeable future — as with NOVUS. Not enough when Luma must serve many stacks.

3 Separate native library per framework Control: High Speed: Slow / costly

Build and maintain parallel React, Angular, and Vue component libraries that share only the tokens. Full control everywhere — at multiplied cost.

Design tokens → React lib Angular lib Vue lib → All apps
Strengths
  • Idiomatic and fully controlled in every framework.
  • No shared-runtime constraints.
Limits
  • N× the build and maintenance.
  • Every fix repeated; stacks drift apart.
  • Needs dedicated teams and budget.

Best when: a large org with funded, dedicated teams per framework and a tolerance for duplicate work.

4 Web-component core + thin framework wrappers Recommended Control: High Speed: Medium

Build each component once as a standards-based web component on top of tokens, then ship thin per-framework wrappers and one Storybook. One codebase, full control, native in every framework.

Design tokens → Web components → Thin wrappers → React / Angular / Vue / jQuery / .NET / HTML
Strengths
  • One implementation — one fix ships everywhere.
  • Full control of markup, design, and behaviour.
  • Framework-agnostic and future-proof.
Limits
  • Up-front setup (tooling + wrappers).
  • Team needs web-component skills.
  • SSR and a few edge cases need care.

Best when: two or more frameworks are in play and you want to own the design — exactly Luma’s situation. See Proposed Approach.

5 Tokens + CSS / utility framework (style-only) Control: Visual only Speed: Fast

Ship design tokens plus a CSS layer (utility classes or SCSS, e.g. Tailwind) and no JavaScript components. Teams write their own markup using your classes.

Design tokens → CSS utilities / classes → Apps build their own markup
Strengths
  • Framework-agnostic and tiny; no JS lock-in.
  • Strong, cheap visual consistency.
Limits
  • No shared behaviour or accessibility.
  • Each team rebuilds interactive components.

Best when: mostly static or content UI, or you only need visual consistency — often paired with another option.

6 Headless components + your own styling Control: High Speed: Medium

A “headless” component is one that ships behaviour but no looks — it handles the hard logic (open/close, keyboard navigation, focus trapping, accessibility) but renders plain, unstyled HTML. You bring 100% of the visual design yourself.

Think of a dropdown: the headless library makes it open on click, move selection with arrow keys, and announce to screen readers correctly — but it looks like raw HTML until you style it. Popular ones: Radix UI, Headless UI, Ark UI. You combine that behaviour with your tokens and your own CSS.

Design tokens + Headless library (behaviour only, no styles) + Your CSS / design → Apps
Strengths
  • The hard part — accessibility & interaction — is done for you.
  • Total visual freedom; the look is entirely yours.
  • Lighter than a full component library.
Limits
  • Most headless libraries are React-only (some support a few others).
  • You still write and maintain all the styling yourself.
  • Doesn’t solve cross-framework on its own.

Best when: a mostly-React org that wants accessibility/behaviour handled but insists on its own unique look. Not a fit for Luma’s many non-React stacks.

7 Buy a commercial component suite Control: Low–Med Speed: Fast

Pay for a ready-made, paid component library from a vendor that already supports many frameworks, and apply your brand theme on top. Instead of building components, you buy a licence and use theirs.

Examples: Syncfusion, AG Grid, or a fuller Telerik / Kendo licence. They ship polished, heavy widgets — data grids, schedulers, charts — for React, Angular, Vue, etc. on day one.

Not a different architecture from Option 1. Luma already follows this pattern with Kendo. Option 7 means buying or switching to another vendor (or adding one just for Grid / Gantt) — still theme-on-vendor, still the same limits on redesigning components.

Design tokens + Paid vendor components (Syncfusion, AG Grid…) + Your theme → All apps
Strengths
  • Heavy data widgets across many frameworks immediately.
  • Vendor handles support, bug fixes, and updates.
  • No component build effort from us.
Limits
  • Ongoing licence cost (often per developer / per year).
  • Same ceiling as Option 1 — we can’t change their markup or behaviour.
  • Locked into the vendor’s roadmap and pricing.

Best when: you urgently need complex widgets (grids, schedulers) across frameworks and are happy to live within a vendor’s design. Good as a complement for heavy components, not as the whole design system.

Option 1 vs Option 7 — same pattern, different choice

Same architecture: your brand (tokens + theme) on top of someone else’s components. You skin them; the vendor owns markup and behaviour. That pattern is listed twice because the business decision is different:

Kendo is a commercial suite (Telerik). Option 7 is not a different technical path — it is “stay on this path but with another vendor or an extra licence for grids/schedulers.” Option 4 is the path that actually changes the architecture.

Why one codebase?

The cost of maintaining a design system grows linearly with the number of framework-specific forks. One codebase keeps the cost flat.

The pain of framework-locked design systems

A typical “React design system” that also wants to support Angular and Vue ends up with three parallel implementations of every component. That means:

  • 3× the maintenance. Every bug fix, every accessibility improvement, every new variant has to be applied to React, Angular, and Vue versions.
  • Visual drift. Components look slightly different over time as changes land in one version but not another.
  • Inconsistent behaviour. Focus management, keyboard shortcuts, and animations diverge between framework implementations.
  • Hiring tax. The design system team needs deep expertise in every framework it supports.
  • Slow rollouts. A new Tabler icon set takes one PR per framework instead of one PR period.

What changes with web components

Framework-locked (one per stack)

  • 3+ component implementations to maintain
  • Bug fixes applied 3+ times
  • Accessibility audited 3+ times
  • Visual drift between framework versions
  • Tied to each framework’s major-version upgrades

Single web-component codebase

  • One component, every framework
  • Bug fixed once, picked up everywhere
  • Accessibility audited once
  • Visual parity by construction
  • Survives framework version churn
The pragmatic exception
Framework wrappers are still maintained — but they are thin (typically 20–50 lines of code) and largely auto-generated. They wrap the component, they don’t re-implement it.

1 Foundation

Layer 1 — Design Tokens

Tokens are the design decisions (colour, spacing, type, radius, motion) expressed as data, not code. One JSON source → many platform outputs.

The source: one JSON file (or a few)

tokens/color.json Source of truth 
{
  "color": {
    "brand": {
      "primary":   { "value": "#d85614" },
      "primary-soft": { "value": "rgba(216, 86, 20, 0.06)" }
    },
    "text": {
      "primary":   { "value": "#111827" },
      "muted":     { "value": "#6b7280" }
    }
  }
}

This format is the W3C Design Tokens Community Group standard — tooling-neutral and now adopted by Figma, Style Dictionary, and major design tools.

2 Components

What is a Web Component?

Web Components is a set of browser standards (Custom Elements, Shadow DOM, HTML Templates) that lets you define new HTML tags with encapsulated styles and behaviour. Once registered, the browser treats them as first-class HTML.

Supported natively in every major browser since 2018 — no polyfill needed for any browser an enterprise actively targets.

A minimal Luma Button as a Lit component

packages/components/src/button/luma-button.ts Lit 
import { LitElement, html, css } from "lit";
import { customElement, property } from "lit/decorators.js";

@customElement("luma-button")
export class LumaButton extends LitElement {
  static styles = css`
    :host { display: inline-block; }
    button {
      background: var(--color-brand-primary);
      color: #fff;
      border: none;
      border-radius: 6px;
      padding: 0.55rem 1.1rem;
      font: inherit;
      cursor: pointer;
    }
    button:disabled { opacity: 0.5; cursor: not-allowed; }
  `;

  @property({ type: String }) variant: "solid" | "outline" = "solid";
  @property({ type: Boolean }) disabled = false;

  render() {
    return html`<button ?disabled=${this.disabled}><slot></slot></button>`;
  }
}

Using it — identical in every framework

Once luma-button is registered, this is valid HTML everywhere:

HTML — works in any context Native 
<luma-button variant="outline">Save</luma-button>

Why Shadow DOM matters

Each web component lives in its own Shadow DOM — an encapsulated subtree the host page’s CSS can’t leak into. This is what makes the component look the same whether it’s embedded in a Tailwind app, a Bootstrap app, or a legacy jQuery + custom CSS app. The host can only style what you explicitly expose (via CSS custom properties or ::part()).

Design tokens + Shadow DOM = consistent look
Inside the shadow DOM, CSS variables defined on the document (:root) are inherited — so the design tokens from Layer 1 reach every component automatically. Encapsulation without isolation.
3 Wrappers

Layer 3 — Framework Wrappers

Same component, idiomatic API per framework. Most wrappers are 20–50 lines and largely auto-generated.

Why wrappers exist at all

React, Angular, and Vue can each technically use a web component directly (<luma-button> just works as HTML). But each framework has quirks that make the bare API awkward:

  • React (before v19) couldn’t pass complex props or listen to custom events idiomatically.
  • Angular needs CUSTOM_ELEMENTS_SCHEMA and benefits from typed inputs / outputs.
  • Vue works out of the box but a typed wrapper gives better DX.
  • Blazor (.NET) needs a small JS interop shim.

The framework matrix

Framework How it consumes
React 19+ Native: <luma-button variant="outline" /> works with refs, props, events. Optional thin wrapper for type safety.
React 18 and below @lit/react or Stencil-generated wrapper (auto-generated).
Angular CUSTOM_ELEMENTS_SCHEMA + thin directive for typed I/O (~20 lines per component, auto-generatable).
Vue 3 Native — web components are first-class in Vue 3. Optional typed wrapper.
jQuery / plain JS Just use the HTML tag; $("luma-button").attr("variant", "outline") — no wrapper needed.
Blazor (.NET) Tag in .razor markup; JS interop for events (~30 lines C# wrapper per component).
Plain HTML Just the tag. One <script> to register elements.

Why this is a big deal
One docs site is the difference between Luma design system being a “React design system with Angular notes” and a real cross-framework design system. The current Luma design system Storybook already does the heavy lifting — the upgrade is to switch the renderer to @storybook/web-components and add multi-framework code snippets per story.

Stencil vs Lit

Two well-supported choices for authoring web components in 2026. They lead to slightly different team workflows; both are production-grade.

Recommendation for a Luma-scale design system

Stencil — if you need React/Angular/Vue wrappers without writing them
For a design system that explicitly wants to ship typed React, Angular, and Vue packages with minimum maintenance overhead, Stencil wins because it auto-generates those wrappers from the same component source. Ionic Framework’s entire UI is built this way — 100+ components shipping to every framework from one codebase. Note: Stencil’s Vue output has historically been less mature than its React and Angular targets — validate in a spike.
Lit — if you want the smallest, most standards-pure stack
Lit is the right call when most of your consumers are modern React (19+), Vue, or plain HTML, and you want minimal tooling. Adobe Spectrum, Shoelace, and Material Web are all Lit-based. Wrappers can be auto-generated via @custom-elements-manifest/analyzer + community tools when you need them.

Honourable mentions

  • FAST (Microsoft) — powers Fluent UI Web Components (distinct from Fluent UI v9, which is React-first). Powerful, slightly heavier.
  • Vanilla Custom Elements — absolutely valid for a small library, but you’ll re-invent reactivity and DX. Not recommended for 50+ components.

Phased rollout for Luma

Today the Luma design system is a shared theme on Kendo UI across React, Angular, Web Components, jQuery, Vue, and .NET — not yet our own component implementations. The quickest path to true framework-agnostic components is incremental — never a big-bang rewrite.

Phase What ships
1. Tokens first Extract Luma design system SCSS variables to Style Dictionary; publish @luma/tokens with CSS, SCSS, and JS outputs. Existing apps adopt with no other changes.
2. Pick the tool Build one component (e.g. Button) twice — once with Stencil, once with Lit — on a feature branch. Pick the winner based on team feedback.
3. First 3 components Build Button, Input, and Card as web components. Ship @luma/components + @luma/react + @luma/angular. New consuming apps start using them; existing apps unaffected.
4. Vue + Blazor + jQuery Add Vue wrapper (small), Blazor wrapper (NuGet), and a CDN bundle for jQuery / plain HTML consumers. Now the system is genuinely cross-framework.
5. Migrate the rest Migrate remaining catalogue components (Scenario A: ~43 as web components). Grid, Gantt, and the Tree family are out of v1 scope — not part of the cross-framework package until a later phase or commercial alternative.

What you keep along the way

  • The Storybook. Switch the renderer to web-components; all existing docs, theming, and the AI Assistant continue to work.
  • The brand and visual language. Tokens are the foundation — not the components — so Luma design system looks identical from day one.
  • Honest scope on heavy UI. Grid, Gantt, and Tree are not in v1. React- only wrappers do not satisfy Vue, Blazor, or jQuery; follow-on work or third-party grids are a separate decision.

Decision points

  • End of Phase 1: have we extracted tokens cleanly, and do consuming apps see no regressions when they upgrade?
  • End of Phase 2: Stencil or Lit — one decision, written down.
  • End of Phase 3: are the React and Angular wrappers idiomatic enough that consuming-team feedback is positive?
  • End of Phase 5: are 43 catalogue components available cross-framework? What is the plan for Grid, Gantt, and Tree (rebuild, commercial lib, or per-stack)?
The pragmatic point
You don’t need to commit to all five phases upfront. Phase 1 (tokens) alone is worth doing regardless — it’s the cheapest win in a design system. Each subsequent phase reads the data from the previous one and decides whether to continue.

Trade-offs & gotchas

Going framework-agnostic is the right architecture, but it has real costs. Knowing them up front prevents surprises.

Where web components are weaker

  • SSR is harder. Declarative Shadow DOM (the modern fix) works in modern browsers but tooling is still catching up. If your apps are heavily SSR (Next.js, SvelteKit), do a small spike before committing.
  • Form-associated custom elements. Native form integration (value, required, validation) works via the ElementInternals API — supported in current Chrome, Firefox, and Safari 16.4+; plan extra QA if you must support older iOS / embedded browsers.
  • Theming via Shadow DOM. Host pages can’t reach inside the shadow DOM with arbitrary CSS. You must explicitly expose theming hooks (CSS custom properties, ::part()). This is actually a feature — but it’s a discipline.
  • Bundle cost. Lit adds ~5 KB; Stencil’s runtime is ~3 KB tree-shaken. Negligible for any real app, but not zero.

Where it costs more than a single-framework system

  • Initial setup. Setting up tokens + Stencil/Lit + wrappers carries an up-front investment that a pure React design system doesn’t pay.
  • Two skill sets. The team needs comfort with both framework-native authoring and the web-component layer.
  • Build complexity. A monorepo + multi-target build pipeline is more moving parts than a single React library.
  • Cross-framework integration testing. The component is written once, but each wrapper (React, Angular, Vue, Blazor) still needs smoke tests in real apps. Budget ongoing QA effort — not a one-time migration cost.

Framework-specific notes

Framework Honest assessment
React 19+ First-class support. Pre-19 needs @lit/react or generated wrapper.
Angular Excellent — web components are a stated supported integration point. Needs CUSTOM_ELEMENTS_SCHEMA in modules.
Vue 3 Easiest of all — native, just works.
jQuery Works fine for reading / writing attributes and listening to events. No two-way binding (jQuery doesn’t have it natively), but selectors and events behave normally.
Blazor (.NET) Works via standard HTML interop. Two-way binding to component properties needs a small JS interop layer.
Server-side WebForms / older .NET Web components render as plain HTML — serve them like any other tag. Interaction is JS-only.
Honest rule of thumb
If more than 80% of your consumers are one framework forever, multi-framework architecture is over-investment. If you have two or more framework consumers today (we already do — React, Angular, Web Components), the maths flips and a unified design system pays back relatively quickly.
