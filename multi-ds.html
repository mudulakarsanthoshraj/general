Summary

The Arcadis Design System is Arcadis Digital Intelligence’s shared component library (Kendo UI, React, Angular, Web Components). Product teams also use Vue, jQuery, and .NET. This proposal asks for a measured, phased path to one framework-agnostic codebase — not a big-bang rewrite.

Scope of this Ask

What this proposal requests now — and what it deliberately does not commit to.

Approving now

Why We Need This

The design system already serves multiple frameworks. Without a single component source, cost and inconsistency grow with every new adopting team.

Arcadis product teams consume Arcadis Design System from React, Angular, and Web Components today. Additional stacks (Vue, jQuery, .NET / Blazor) appear in roadmaps. Each stack currently depends on Kendo-themed paths and documentation that must stay aligned by hand.

  • A bug fix or accessibility improvement in one surface may need repeating across framework-specific examples and wrappers.
  • Brand updates (colour, spacing, type) touch SCSS and docs in multiple places.
  • Consuming teams see small visual differences between stacks — undermining “one Arcadis experience.”
  • Onboarding a new framework (e.g. Vue) means re-proving patterns instead of reusing a published package.

The trigger
We are past the point where a single-framework design system is cheaper. With two or more active consumer frameworks, industry practice is one web-component core plus thin wrappers — not three parallel implementations.

Proposed Approach

Four layers, one repo. Technical detail lives in the architecture briefing; here is the executive view.

Layer What it is Delivers
1. Design tokens JSON source of truth (colour, spacing, type) CSS, SCSS, JS for every platform
2. Web components One implementation per component (Lit or Stencil) Same HTML tag in every framework
3. Framework wrappers Thin React / Angular / Vue / Blazor adapters Idiomatic APIs; no re-implementation
4. Documentation One Storybook Single place to learn and QA components

Example: <arcadis-button variant="outline"> is authored once. React, Angular, Vue, jQuery, and Blazor apps all consume the same element — wrappers only translate framework conventions.

Proof it is mainstream
Adobe Spectrum, IBM Carbon, Ionic, Shoelace, and Google Material Web all ship this way. See the briefing for comparisons and code samples.

Alternatives Considered

Four other paths. We recommend the phased web-component approach.

Option Summary Why it falls short
A. Status quo Keep Kendo + per-framework Storybook paths as today. Lowest short-term effort; drift and duplicate work grow with adoption. Each new framework multiplies maintenance.
B. Separate design system per framework Dedicated React DS, Angular DS, Vue DS teams. ~3× component work, ~3× accessibility audits. Hard to keep visual parity; expensive at scale.
C. Buy a commercial design system License a vendor SaaS / component library. Often $10K–$50K+ per year, generic styling, weak fit for Arcadis Design System-specific Kendo + Tabler patterns.
D. Tokens + web components (recommended) Phased migration; Kendo paths coexist during transition. Up-front setup cost, but one fix ships everywhere. Matches how the design system is already consumed across stacks.

Cost of Inaction

What we pay if we keep the current multi-path model.

  • Duplicate engineering: Every component change (variant, a11y fix, token rename) is repeated across framework surfaces — roughly proportional to the number of active stacks.
  • Inconsistent UX: Teams and auditors notice when Angular and React apps don’t match; rework lands on the design system team anyway.
  • Slower adoption: New product lines on Vue or .NET wait for “their” Arcadis Design System path instead of installing a published package.
  • Documentation debt: Storybook and Slack support scale with questions × frameworks, not with a single source of truth.
Directional, not precise
We are not claiming a single dollar ROI figure without baseline data. Phase 1 gives measurable token adoption; Phase 3 gives before/after maintenance time on three pilot components.

The idea in one paragraph

Treat the design system as four layered packages: design tokens (JSON) → web components (the actual UI) → thin framework wrappers (React / Angular / Vue) → one Storybook documenting all of them. Web Components are a native browser standard, so the same <arcadis-button> tag works identically in every framework. Wrappers exist purely to give each framework an idiomatic API (typed props in React, [(ngModel)] in Angular, etc.) — not to re-implement the component.

Who already does this (proof it works)

  • Adobe Spectrum Web Components — Lit-based, React + plain
  • Fluent UI Web Components (Microsoft, FAST) — cross-framework; note Fluent UI v9 itself is React-first
  • IBM Carbon — React, Angular, Vue, Web Components
  • Salesforce Lightning Web Components — vanilla + LWC
  • Shoelace / Web Awesome — Lit, every framework
  • Ionic Framework — Stencil; React, Angular, Vue, vanilla
  • Material Web (Google) — Lit, every framework
Bottom line
This isn’t experimental. It’s the architecture every major cross-framework design system has converged on over the last 5 years. The question is which tools, not if it can work.

For management review — scope, cost, timeline, and decision gates — see the Proposal section at the top of this document.

Why one codebase?

The cost of maintaining a design system grows linearly with the number of framework-specific forks. One codebase keeps the cost flat.

The pain of framework-locked design systems

A typical “React design system” that also wants to support Angular and Vue ends up with three parallel implementations of every component. That means:

  • 3× the maintenance. Every bug fix, every accessibility improvement, every new variant has to be applied to React, Angular, and Vue versions.
  • Visual drift. Components look slightly different over time as changes land in one version but not another.
  • Inconsistent behaviour. Focus management, keyboard shortcuts, and animations diverge between framework implementations.
  • Hiring tax. The design system team needs deep expertise in every framework it supports.
  • Slow rollouts. A new Tabler icon set takes one PR per framework instead of one PR period.

What changes with web components

Framework-locked (one per stack)

  • 3+ component implementations to maintain
  • Bug fixes applied 3+ times
  • Accessibility audited 3+ times
  • Visual drift between framework versions
  • Tied to each framework’s major-version upgrades

Single web-component codebase

  • One component, every framework
  • Bug fixed once, picked up everywhere
  • Accessibility audited once
  • Visual parity by construction
  • Survives framework version churn
The pragmatic exception
Framework wrappers are still maintained — but they are thin (typically 20–50 lines of code) and largely auto-generated. They wrap the component, they don’t re-implement it.

Style Dictionary

How a change flows up the stack

Imagine the brand colour changes from #d85614 to #e85a16:

  1. Layer 1: update brand.primary in tokens.json.
  2. Layer 1: Style Dictionary rebuilds — outputs new CSS variables, SCSS, JS, iOS, Android in one command.
  3. Layer 2: Every web component reads var(--color-brand-primary); no component change needed.
  4. Layer 3: Wrappers don’t care; they just forward to the components.
  5. Layer 4: Storybook reflects the change automatically.

One commit, every consuming app updated on next dependency bump.

1 Foundation

Layer 1 — Design Tokens

Tokens are the design decisions (colour, spacing, type, radius, motion) expressed as data, not code. One JSON source → many platform outputs.

The source: one JSON file (or a few)

tokens/color.json Source of truth 
{
  "color": {
    "brand": {
      "primary":   { "value": "#d85614" },
      "primary-soft": { "value": "rgba(216, 86, 20, 0.06)" }
    },
    "text": {
      "primary":   { "value": "#111827" },
      "muted":     { "value": "#6b7280" }
    }
  }
}

This format is the W3C Design Tokens Community Group standard — tooling-neutral and now adopted by Figma, Style Dictionary, and major design tools.

The tool: Style Dictionary

Style Dictionary (open source, Amazon) reads the JSON and emits exactly the formats each platform needs:

Output Consumer Example
CSS variables Web components, plain HTML, Tailwind v4 --color-brand-primary: #d85614;
SCSS variables Legacy stylesheets, Kendo themes $color-brand-primary: #d85614;
JS / TS React Native, JS-in-CSS, design libraries export const colorBrandPrimary = "#d85614";
iOS (Swift) Native iOS apps UIColor.brandPrimary
Android (XML) Native Android apps <color name="brand_primary">#d85614</color>
JSON Figma Tokens plugin, design tooling { "color": { "brand": { ... } } }
Why this matters first
Tokens are the cheapest layer to migrate to first, and they pay back immediately — every app that consumes Arcadis Design System can start using --color-brand-primary today, even before any component is rewritten as a web component.
2 Components

Layer 2 — Web Components

The single component implementation that every framework consumes. Native to the browser; no virtual DOM, no framework lock-in.

What is a Web Component?

Web Components is a set of browser standards (Custom Elements, Shadow DOM, HTML Templates) that lets you define new HTML tags with encapsulated styles and behaviour. Once registered, the browser treats them as first-class HTML.

Supported natively in every major browser since 2018 — no polyfill needed for any browser an enterprise actively targets.

A minimal Arcadis Button as a Lit component

packages/components/src/button/arcadis-button.ts Lit 
import { LitElement, html, css } from "lit";
import { customElement, property } from "lit/decorators.js";

@customElement("arcadis-button")
export class ArcadisButton extends LitElement {
  static styles = css`
    :host { display: inline-block; }
    button {
      background: var(--color-brand-primary);
      color: #fff;
      border: none;
      border-radius: 6px;
      padding: 0.55rem 1.1rem;
      font: inherit;
      cursor: pointer;
    }
    button:disabled { opacity: 0.5; cursor: not-allowed; }
  `;

  @property({ type: String }) variant: "solid" | "outline" = "solid";
  @property({ type: Boolean }) disabled = false;

  render() {
    return html`<button ?disabled=${this.disabled}><slot></slot></button>`;
  }
}

Using it — identical in every framework

Once arcadis-button is registered, this is valid HTML everywhere:

HTML — works in any context Native 
<arcadis-button variant="outline">Save</arcadis-button>

Why Shadow DOM matters

Each web component lives in its own Shadow DOM — an encapsulated subtree the host page’s CSS can’t leak into. This is what makes the component look the same whether it’s embedded in a Tailwind app, a Bootstrap app, or a legacy jQuery + custom CSS app. The host can only style what you explicitly expose (via CSS custom properties or ::part()).

Design tokens + Shadow DOM = consistent look
Inside the shadow DOM, CSS variables defined on the document (:root) are inherited — so the design tokens from Layer 1 reach every component automatically. Encapsulation without isolation.
3 Wrappers

Layer 3 — Framework Wrappers

Same component, idiomatic API per framework. Most wrappers are 20–50 lines and largely auto-generated.

Why wrappers exist at all

React, Angular, and Vue can each technically use a web component directly (<arcadis-button> just works as HTML). But each framework has quirks that make the bare API awkward:

  • React (before v19) couldn’t pass complex props or listen to custom events idiomatically.
  • Angular needs CUSTOM_ELEMENTS_SCHEMA and benefits from typed inputs / outputs.
  • Vue works out of the box but a typed wrapper gives better DX.
  • Blazor (.NET) needs a small JS interop shim.

The framework matrix

Why this is a big deal
One docs site is the difference between Arcadis Design System being a “React design system with Angular notes” and a real cross-framework design system. The current Arcadis Design System Storybook already does the heavy lifting — the upgrade is to switch the renderer to @storybook/web-components and add multi-framework code snippets per story.

Stencil vs Lit

Two well-supported choices for authoring web components in 2026. They lead to slightly different team workflows; both are production-grade.

  Lit Stencil
Type Runtime library (~5 KB) Compiler that emits standard web components
Maintainer Google (Polymer team) Ionic
Authoring style TypeScript classes + tagged-template literals TypeScript classes + JSX
Build step Optional (TS or plain JS) Required (Stencil compiler)
Framework wrappers Manual or @lit/react Auto-generated React / Angular / Vue wrappers
SSR / hydration Lit SSR (Declarative Shadow DOM) Built-in (prerendering / hydration)
Bundle output Per-component ESM Tree-shakable bundle per framework target
Best for Small footprint, full control, modern apps Big component library shipping to multiple framework consumers

Recommendation for a Arcadis-scale system

Stencil — if you need React/Angular/Vue wrappers without writing them
For a design system that explicitly wants to ship typed React, Angular, and Vue packages with minimum maintenance overhead, Stencil wins because it auto-generates those wrappers from the same component source. Ionic Framework’s entire UI is built this way — 100+ components shipping to every framework from one codebase. Note: Stencil’s Vue output has historically been less mature than its React and Angular targets — validate in a spike.
Lit — if you want the smallest, most standards-pure stack
Lit is the right call when most of your consumers are modern React (19+), Vue, or plain HTML, and you want minimal tooling. Adobe Spectrum, Shoelace, and Material Web are all Lit-based. Wrappers can be auto-generated via @custom-elements-manifest/analyzer + community tools when you need them.

Honourable mentions

  • FAST (Microsoft) — powers Fluent UI Web Components (distinct from Fluent UI v9, which is React-first). Powerful, slightly heavier.
  • Vanilla Custom Elements — absolutely valid for a small library, but you’ll re-invent reactivity and DX. Not recommended for 50+ components.

Trade-offs & gotchas

Going framework-agnostic is the right architecture, but it has real costs. Knowing them up front prevents surprises.

Where web components are weaker

  • SSR is harder. Declarative Shadow DOM (the modern fix) works in modern browsers but tooling is still catching up. If your apps are heavily SSR (Next.js, SvelteKit), do a small spike before committing.
  • Form-associated custom elements. Native form integration (value, required, validation) works via the ElementInternals API — supported in current Chrome, Firefox, and Safari 16.4+; plan extra QA if you must support older iOS / embedded browsers.
  • Theming via Shadow DOM. Host pages can’t reach inside the shadow DOM with arbitrary CSS. You must explicitly expose theming hooks (CSS custom properties, ::part()). This is actually a feature — but it’s a discipline.
  • Bundle cost. Lit adds ~5 KB; Stencil’s runtime is ~3 KB tree-shaken. Negligible for any real app, but not zero.

Where it costs more than a single-framework system

  • Initial setup. Setting up tokens + Stencil/Lit + wrappers carries an up-front investment that a pure React design system doesn’t pay.
  • Two skill sets. The team needs comfort with both framework-native authoring and the web-component layer.
  • Build complexity. A monorepo + multi-target build pipeline is more moving parts than a single React library.
  • Cross-framework integration testing. The component is written once, but each wrapper (React, Angular, Vue, Blazor) still needs smoke tests in real apps. Budget ongoing QA effort — not a one-time migration cost.

Framework-specific notes

Framework Honest assessment
React 19+ First-class support. Pre-19 needs @lit/react or generated wrapper.
Angular Excellent — web components are a stated supported integration point. Needs CUSTOM_ELEMENTS_SCHEMA in modules.
Vue 3 Easiest of all — native, just works.
jQuery Works fine for reading / writing attributes and listening to events. No two-way binding (jQuery doesn’t have it natively), but selectors and events behave normally.
Blazor (.NET) Works via standard HTML interop. Two-way binding to component properties needs a small JS interop layer.
Server-side WebForms / older .NET Web components render as plain HTML — serve them like any other tag. Interaction is JS-only.
Honest rule of thumb
If more than 80% of your consumers are one framework forever, multi-framework architecture is over-investment. If you have two or more framework consumers today (we already do — React, Angular, Web Components), the maths flips and a unified design system pays back relatively quickly.

Phased rollout for the design system

The Arcadis Design System already supports React, Angular, and Web Components on top of Kendo UI. The quickest path to a true framework-agnostic system is incremental — never a big-bang rewrite.

The five phases

Phase What ships Indicative effort
1. Tokens first Extract Arcadis Design System SCSS variables to Style Dictionary; publish @arcadis/tokens with CSS, SCSS, and JS outputs. Existing apps adopt with no other changes. S
2. Pick the tool Build one component (e.g. Button) twice — once with Stencil, once with Lit — on a feature branch. Pick the winner based on team feedback. XS
3. First 3 components Build Button, Input, and Card as web components. Ship @arcadis/components + @arcadis/react + @arcadis/angular. New consuming apps start using them; existing apps unaffected. M–L
4. Vue + Blazor + jQuery Add Vue wrapper (small), Blazor wrapper (NuGet), and a CDN bundle for jQuery / plain HTML consumers. Now the system is genuinely cross-framework. S
5. Migrate the rest Migrate remaining catalogue components (Scenario A: ~43 as web components). Grid, Gantt, and the Tree family are out of v1 scope — not part of the cross-framework package until a later phase or commercial alternative. XL

What you keep along the way

  • The Storybook. Switch the renderer to web-components; all existing docs, theming, and the AI Assistant continue to work.
  • The brand and visual language. Tokens are the foundation — not the components — so Arcadis Design System looks identical from day one.
  • Honest scope on heavy UI. Grid, Gantt, and Tree are not in v1. React- only wrappers do not satisfy Vue, Blazor, or jQuery; follow-on work or third-party grids are a separate decision.

Decision points

  • End of Phase 1: have we extracted tokens cleanly, and do consuming apps see no regressions when they upgrade?
  • End of Phase 2: Stencil or Lit — one decision, written down.
  • End of Phase 3: are the React and Angular wrappers idiomatic enough that consuming-team feedback is positive?
  • End of Phase 5: are 43 catalogue components available cross-framework? What is the plan for Grid, Gantt, and Tree (rebuild, commercial lib, or per-stack)?
The pragmatic point
You don’t need to commit to all five phases upfront. Phase 1 (tokens) alone is worth doing regardless — it’s the cheapest win in a design system. Each subsequent phase reads the data from the previous one and decides whether to continue.
