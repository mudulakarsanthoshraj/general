Project Overview

BDB API Backend is an ASP.NET Core Web API that powers the BDB (Bouw Database – Construction Database) platform. It serves as the central data layer for all frontend clients, providing construction cost data, market analytics, forecasts, user management, and document export capabilities.

What is BDB?

BDB (Bouw Database – Construction Database) is a comprehensive platform for the Dutch construction industry. The backend API is the single source of truth for all data that the frontend (and external clients) consume. It provides:

  • Monthly Figures (Maandcijfers): Construction cost indices (labor, material, equipment) served as structured data with historical time series and econometric forecasts
  • Market Indicators (Marktindicatoren): Macro construction market trend data and analytics
  • Aspect Ratios / Gi (Kengetallen): Cost breakdown model (Element Clusters, Stabu Chapters, Product Groups) across project types and regions
  • Parametric Elements: Searchable catalog of standardized building components with unit costs
  • Reference Projects (Referentieprojecten): Completed project budgets used as benchmarks
  • Ad-Hoc Projects: User-created projects for custom budget tracking and scenario analysis
  • WOV Index: Woningbouw, Onderhoud en Verduurzamingsindex – residential construction sustainability index
  • Design Key Cost Figures (DFKK): Design-phase cost estimation model (construction, investment, operational costs)
  • Excel Exports: Server-side generation of reports using Aspose.Cells for all data modules
  • Subscription Management: Access control based on company subscriptions and user roles

Business Context

The backend serves construction companies, cost consultants, project managers, and real estate developers in the Netherlands via the BDB frontend. Key responsibilities of the API include:

  • Data Aggregation: Querying and processing large SQL Server datasets (cube queries via DAX, stored procedures) into consumable API responses
  • Access Control: Enforcing subscription-based and role-based access so users only receive data they are entitled to
  • File Processing: Receiving Excel uploads from admins, parsing and validating them, and persisting to the database
  • Report Generation: Building multi-sheet Excel and PDF reports using Aspose.Cells and returning them as downloadable blobs
  • Integration: Communicating with Azure Blob Storage, Azure Key Vault, ECB exchange rate API, and Microsoft Graph for email

User Roles & Access

Standard User ("user")

  • Authenticated via Azure AD B2C JWT token
  • Access to API endpoints restricted by subscription modules
  • Can read data, generate reports, manage personal presets
  • Can create and manage their own ad-hoc projects
  • Cannot access upload or admin-only endpoints
  • Date ranges restricted by subscription tier

Admin User ("admin")

  • All standard user capabilities
  • Access to upload endpoints (reference budgets, forecast figures, WOV)
  • Bypasses subscription date/module restrictions
  • Can publish data in bulk across the system
  • Can trigger workflow actions (publish, approve)
  • Full access to all data modules regardless of subscription
Key API Capabilities
  • Multi-tenant architecture — all requests are scoped to a clientId (company)
  • Role-based authorization (admin / user) enforced at controller level
  • Subscription-gated data access with date-range enforcement
  • Server-side Excel report generation via Aspose.Cells (licensed)
  • Excel file import pipeline with validation, parsing, and persistence
  • API Key authentication for external client integrations (/api/BDBClient)
  • Background task queue for long-running operations (publish, bulk upload)
  • Azure Blob Storage for uploaded files and generated exports
  • Comprehensive action logging for audit trails

Setup & Installation

Step-by-step guide to get the BDB API running on your local machine.

Prerequisites

Required Software
  • .NET 8 SDK: Download from https://dotnet.microsoft.com/download
  • SQL Server: 2019 or later (or SQL Server LocalDB for dev)
  • Azure CLI: Required for Key Vault access during local dev (see installation below)
  • Git: For cloning the repository
  • IDE: Visual Studio 2022 17.8+ or Rider recommended; VS Code works too

Install Azure CLI

The Azure CLI is required for local development because the API uses DefaultAzureCredential to authenticate with Azure Key Vault and load secrets. Without it, the app cannot start locally.

macOS Installation

# Install via Homebrew
brew update
brew install azure-cli

# Verify installation
az --version

Windows Installation

# Download and run the MSI installer from:
https://aka.ms/installazurecliwindows

# Or via winget:
winget install -e --id Microsoft.AzureCLI

# Verify installation
az --version

After installation, authenticate with your Azure account: 

az login

# Verify you're logged into the correct tenant
az account show
Why Azure CLI is Required

The BDB API uses DefaultAzureCredential in Program.cs to authenticate with Azure Key Vault. In local development, this credential chain checks for:

  1. Environment variables (service principal credentials)
  2. Azure CLI credentials (from az login) ← Most common for local dev
  3. Managed Identity (only works in Azure)
  4. Visual Studio or VS Code credentials

Without az login, the app will fail to start with an authentication error when trying to load secrets from Key Vault.

Verify all required tools are installed:

dotnet --version          # Should output 8.x.x
az --version              # Should show Azure CLI version (e.g., 2.x.x)
git --version             # Should show Git version

Architecture

Understanding the solution's layered structure, project responsibilities, and design decisions.

High-Level Architecture

The solution follows Clean Architecture (Ports & Adapters) with four distinct layers:

Frontend / External Client → BdbAPI (Presentation) → Application (Business Logic)
Application → Domain (Entities) & Infrastructure (Data Access) → SQL Server / Azure
Why This Layered Architecture?
  • Separation of Concerns: Controllers know nothing about the database; services know nothing about HTTP — each layer has one job
  • Testability: Services are injected via interfaces → easy to mock in unit tests without a real DB
  • Maintainability: Changing the database (e.g., switching from EF to Dapper) only affects Infrastructure — Application and Domain are unchanged
  • Scalability: New features are added as new services + controllers following the same pattern — no "big ball of mud"

Request Lifecycle

1
HTTP Request Arrives
Frontend sends request with Authorization: Bearer {token} and ?clientId=123
2
Middleware Pipeline
CORS → ExceptionMiddleware → ApiKeyMiddleware → Authentication → Authorization
3
Controller
Controller method extracts params, calls service method
4
Service (Business Logic)
Service validates, applies business rules, calls repositories or cube client
5
Repository / DB
EF Core or stored procedure fetches/writes data from SQL Server
6
HTTP Response
Data returned as JSON (camelCase) or as file blob for exports

Dependency Flow (Clean Architecture)

Inner layers know nothing about outer layers. Dependencies always point inward:

Layer Dependency Rules
  • Domain — zero dependencies. Pure C# classes (entities, enums, base interfaces). No NuGet packages.
  • Application — depends on Domain only. Contains service interfaces. Infrastructure details injected via constructor DI.
  • Infrastructure — depends on Domain + Application interfaces. Implements repositories, DB context, HTTP clients.
  • BdbAPI (Presentation) — depends on Application interfaces. Controllers call services; never calls repositories directly.

Dependency Injection Pattern

All dependencies are registered in the two DependencyInjection.cs files — one in Application, one in Infrastructure — and both are called from Startup.cs: 

// Startup.cs - ConfigureServices()
services.AddInfrastructure(Configuration);  // DB context, repositories, HTTP clients
services.AddApplication(Configuration);     // All business services

// Application/DependencyInjection.cs (simplified)
services.AddScoped<IProfileService, ProfileService>();
services.AddScoped<IReportForecastFiguresService, ReportForecastFiguresService>();
services.AddScoped<IParametricNodeService, ParametricNodeService>();
// ... 30+ more service registrations

// Infrastructure/DependencyInjection.cs (simplified)
services.AddDbContext<BdbPlatformDbContext>(options =>
    options.UseSqlServer(connectionString));
services.AddScoped<IUserRepository, UserRepository>();
services.AddScoped<ICubeRepository, CubeRepository>();
// ... all repository registrations

Key Architectural Patterns

1. Repository Pattern

All data access goes through repositories. Services never use BdbPlatformDbContext directly — they inject repository interfaces. Why? Repositories can be mocked in service unit tests, and all DB queries are in one place.

  • IGenericDatabaseRepo<T> — generic CRUD for any entity
  • IUserRepository, IReferenceBudgetRepository, etc. — entity-specific repositories with custom queries
  • IStoredProcedures — wraps all stored procedure calls
  • ICubeRepository — executes DAX queries against Analysis Services cube

2. Token Provider Pattern

ITokenProvider (implemented by TokenProvider) reads claims from the current JWT token (email, user ID, company ID). Services inject this to identify the calling user without touching HttpContext directly. 

// TokenProvider extracts claims from the current HTTP request's JWT
public interface ITokenProvider {
    string GetUserEmailId();
    int GetClientId();     // Extracted from query param ?clientId=
}

3. Service Scoping

  • Scoped (AddScoped): Most services — created once per HTTP request, disposed at end. Used for services accessing DB (e.g., ProfileService, ReportForecastFiguresService)
  • Transient (AddTransient): Lightweight, stateless services — new instance per injection (e.g., ReferenceDatePublishedService, PathProvider)
  • Singleton (AddSingleton): One instance for the app lifetime — used for thread-safe shared resources (e.g., IAzureBlobClient, IBackgroundTaskQueue)

Authentication & Authorization

The API uses Azure AD B2C JWT Bearer tokens for user authentication and enforces role/subscription-based authorization at the controller level.

Authentication Flow

Frontend sends request → Authorization: Bearer {JWT} → ASP.NET Core JWT Middleware
JWT validated against → Azure AD B2C JWKS endpoint → Claims set on HttpContext.User
TokenProvider reads claims → User email + company resolved → Service executes with user context

JWT Bearer Configuration (Startup.cs)

// Startup.cs — ConfigureServices
services.AddAuthentication(JwtBearerDefaults.AuthenticationScheme)
    .AddMicrosoftIdentityWebApi(options => {
        Configuration.Bind("AzureAdB2C", options);
    },
    options => {
        Configuration.Bind("AzureAdB2C", options);
    });

// Required settings in appsettings.json → AzureAdB2C section:
//   Instance, Domain, Tenant, ClientId, SignUpSignInPolicyId
Azure AD B2C Settings
  • Policy: B2C_1A_BDB_SIGNUP_SIGNIN — custom sign-up/sign-in policy
  • Token Lifetime: Controlled by B2C policy (UseTokenLifetime: true)
  • PKCE: Enabled (UsePkce: true) for security
  • Secrets: ClientSecret stored in Azure Key Vault (never in code or config files)

Base Controller & Route Protection

All controllers inherit from BaseApiController which applies [Authorize] globally: 

// BaseApiController.cs — inherited by all controllers
[Authorize]           // Requires valid JWT on every endpoint
[ApiController]
[Route("api/[controller]")]
public class BaseApiController : ControllerBase
{
    // Helper: checks if user has permission for the given date range
    protected bool HasPermissionForDates(
        List<string> userSubscription,
        string userRole,
        DateTime minDate, DateTime maxDate,
        List<DateTime> dates)
    {
        // Admin/operator role → always allowed
        if (AuthUtility.CanAccessByOperator(userRole)) return true;

        // GisHc subscription → can access up to today
        if (AuthUtility.CanAccessedByGisHcBasic(userSubscription)
            && dates.All(d => minDate <= d && d <= DateTime.Now)) return true;

        // GisHcf subscription (with forecast) → can access up to maxDate
        if (AuthUtility.CanAccessedByGisHcfBasic(userSubscription)
            && dates.All(d => minDate <= d && d <= maxDate)) return true;

        return false;
    }
}

Role-Based Authorization

The AuthController provides two endpoints used by the frontend to determine access: 

// GET api/Auth/GetRoles
// Returns: "admin" or "user" (single string)
[AllowAnonymous]
[HttpGet("GetRoles")]
public IActionResult GetUserRole()
{
    var userRole = _authService.GetUserRole();
    return Ok(userRole);
}

// GET api/Auth/GetSubscriptions
// Returns: List<string> of subscription module codes
[AllowAnonymous]
[HttpGet("GetSubscriptions")]
public IActionResult GetUserSubscriptions()
{
    var subscriptions = _authService.GetUserSubscriptions();
    return Ok(subscriptions);
}
How AuthorizationService Works

1. Get email from token: TokenProvider.GetUserEmailId() reads the email claim from the JWT

2. Look up user in DB: Queries UserRepository to find the User record by email

3. Resolve company: Gets the clientId from the query parameter (passed by frontend on every request)

4. Get role: Queries UserRoleRepository to find the user's role for that company — returns "admin" or "user"

5. Get subscriptions: Queries ModuleCompany to find all active module subscriptions for that company

API Key Authentication (External Clients)

The /api/BDBClient route uses a separate API key mechanism (not JWT) for external system integrations: 

// ApiKeyMiddleware.cs — checks X-Api-Key header
// Only applies to /api/BDBClient route
// Key format: Base64(companyId:companyKey)
// Config key: "apikey-company{id}" from Azure Key Vault

// Example header:
X-Api-Key: Base64("42:supersecretkey")
Important Security Notes
  • JWT tokens are validated cryptographically against Azure AD B2C's JWKS — no shared secrets needed
  • API keys for BDBClient are stored in Azure Key Vault — never hardcode in source
  • The [AllowAnonymous] on Auth endpoints is intentional — the frontend calls these immediately after token acquisition to populate user context
  • Admin-only actions are protected by ForbiddenAccessException thrown from AuthorizationService.RequireAdminRole() → caught by ExceptionMiddleware → returns HTTP 403

Controllers & Endpoints

Overview of all API controllers, their route groups, and responsibilities. All routes follow the pattern /api/[ControllerName] and require JWT authorization unless marked [AllowAnonymous].

Controller Design Rules
  • All controllers inherit BaseApiController → automatically get [Authorize] and /api/[controller] routing
  • Controllers are thin — they only validate input, call one service method, and return a result
  • Business logic lives exclusively in the Application layer services
  • JSON responses use camelCase (configured globally in Startup.cs)
  • File responses return FileStreamResult or FileContentResult for Excel exports

Services Layer

The Application layer contains all business logic. Each service encapsulates one feature domain. Services are the only layer that controllers interact with.

Why the Service Layer Pattern?
  • Testability: Services are injected as interfaces → easily mocked in unit tests without a real database
  • Single Responsibility: Each service owns one domain (profile, forecast figures, parametric, etc.)
  • Reusability: The same ProfileService.GetProfileAsync() is called from multiple controllers
  • Maintainability: Business rule changes only require updating one service class
  • Transaction Boundaries: Complex multi-step operations (upload → parse → validate → save → notify) are orchestrated entirely within a service

Service Pattern Example

// ProfileService.cs — typical service pattern
public class ProfileService : IProfileService
{
    private readonly ITokenProvider _tokenProvider;
    private readonly IUserRepository _userRepo;
    private readonly IGenericDatabaseRepo<Person> _personRepo;

    // All dependencies injected — never "new" them inside
    public ProfileService(
        ITokenProvider tokenProvider,
        IUserRepository userRepo,
        IGenericDatabaseRepo<Person> personRepo)
    {
        _tokenProvider = tokenProvider;
        _userRepo = userRepo;
        _personRepo = personRepo;
    }

    public async Task<ProfileViewModel> GetProfileAsync()
    {
        // 1. Identify user from JWT
        var email = _tokenProvider.GetUserEmailId();

        // 2. Load entity from DB via repository
        var user = await _userRepo.GetByEmailAsync(email);

        // 3. Map to ViewModel (never return EF entities directly)
        return new ProfileViewModel {
            FirstName = user.Person.FirstName,
            LastName = user.Person.LastName,
            Email = user.Email,
            CompanyName = user.Company.Name
        };
    }
}

Middleware Pipeline

ASP.NET Core middleware processes every HTTP request in sequence. Understanding the order is critical for debugging authentication and error scenarios.

Pipeline Order

Order matters — middleware runs top-to-bottom on request, bottom-to-top on response:

1
Developer Exception Page
Only in Development. Shows full stack trace for unhandled exceptions.
2
Swagger UI
Only in Development. Serves /swagger endpoint for API exploration.
3
Routing
Matches the incoming URL to a controller endpoint.
4
CORS
Applies the "test" policy: allow any origin, any method, expose Authorization & Content-Disposition headers.
5

Custom global error handler. Catches ForbiddenAccessException → 403, UploadExcelException → 400.
6

Checks X-Api-Key header for /api/BDBClient routes only.
7

Validates JWT Bearer token against Azure AD B2C JWKS.
8
Authorization
Enforces [Authorize] — rejects requests with invalid/missing tokens.

ExceptionMiddleware

Catches specific custom exceptions thrown anywhere in the pipeline and maps them to HTTP status codes. Without this, all exceptions would result in unstructured 500 responses.

Exception HTTP Status When Thrown
ForbiddenAccessException 403 Forbidden User does not have admin role for an admin-only action
UploadExcelException 400 Bad Request Uploaded Excel file fails validation (wrong format, missing columns, bad data)
Any other exception 500 (unhandled) Unexpected error — logged to Application Insights 
// ExceptionMiddleware.cs — catches known exceptions
catch (ForbiddenAccessException ex) {
    context.Response.StatusCode = 403;
    await context.Response.WriteAsJsonAsync(new { error = ex.Message });
}
catch (UploadExcelException ex) {
    context.Response.StatusCode = 400;
    await context.Response.WriteAsJsonAsync(new { error = ex.Description });
}

ApiKeyMiddleware

Only intercepts requests to /api/BDBClient. Validates X-Api-Key header (Base64-encoded companyId:key) against values stored in Azure Key Vault. 

// ApiKeyMiddleware — only for /api/BDBClient
// Header format: X-Api-Key: Base64("companyId:apiKey")

Background Processing & Functions

Long-running tasks run asynchronously via a background task queue. Azure Functions handle scheduled and external-trigger operations.

Background Task Queue

Some operations (e.g., bulk publish of reference budgets, data migration) are too slow to run synchronously within an HTTP request's timeout. The solution uses a producer-consumer queue:

Controller receives request → Enqueues task to IBackgroundTaskQueue → Returns 202 Accepted immediately
BackgroundProcessor (Hosted Service) → Dequeues and executes task → Updates DB status on completion 
// Enqueue a background task from a service
_taskQueue.QueueBackgroundWorkItem(async (cancellationToken) =>
{
    // This runs asynchronously after response is sent
    await ProcessBulkUploadAsync(uploadId, cancellationToken);
});

// BackgroundProcessor (IHostedService) dequeues and runs tasks
while (!cancellationToken.IsCancellationRequested)
{
    var workItem = await _taskQueue.DequeueAsync(cancellationToken);
    await workItem(cancellationToken);
}
Background Task Use Cases
  • Bulk Publish: Publishing 100s of reference budget items in one operation — long DB transaction
  • Excel Processing: Parsing and validating large uploaded Excel files without HTTP timeout
  • Email Notifications: Sending upload confirmation emails via Graph API after upload completes
  • Data Migration: Classic system user migration via migration API

Test Patterns

1. Service Unit Test Pattern

Setup: Mock all repository and infrastructure dependencies. Inject mocks into the real service. Test real business logic.

// ProfileServiceTest.cs — typical service test
[TestClass]
public class ProfileServiceTest
{
    private Mock<ITokenProvider> _mockTokenProvider;
    private Mock<IUserRepository> _mockUserRepo;
    private Mock<IGenericDatabaseRepo<Person>> _mockPersonRepo;
    private ProfileService _service;  // Real service, not mocked

    [TestInitialize]
    public void Setup()
    {
        // Create mocks for all dependencies
        _mockTokenProvider = new Mock<ITokenProvider>();
        _mockUserRepo = new Mock<IUserRepository>();
        _mockPersonRepo = new Mock<IGenericDatabaseRepo<Person>>();

        // Set up default behaviour
        _mockTokenProvider.Setup(t => t.GetUserEmailId())
            .Returns("john.doe@example.com");

        // Inject mocks into REAL service
        _service = new ProfileService(
            _mockTokenProvider.Object,
            _mockUserRepo.Object,
            _mockPersonRepo.Object);
    }

    [TestMethod]
    public async Task GetProfileAsync_ReturnsProfile_WhenUserExists()
    {
        // Arrange: set up mock DB response
        _mockUserRepo.Setup(r => r.GetByEmailAsync("john.doe@example.com"))
            .ReturnsAsync(new User {
                Email = "john.doe@example.com",
                Person = new Person { FirstName = "John", LastName = "Doe" },
                Company = new Company { Name = "TestCorp" }
            });

        // Act: call the real service method
        var result = await _service.GetProfileAsync();

        // Assert: verify output
        Assert.AreEqual("John", result.FirstName);
        Assert.AreEqual("Doe", result.LastName);
        Assert.AreEqual("TestCorp", result.CompanyName);
    }

    [TestMethod]
    public async Task GetProfileAsync_ThrowsException_WhenUserNotFound()
    {
        // Arrange: simulate user not in DB
        _mockUserRepo.Setup(r => r.GetByEmailAsync(It.IsAny<string>()))
            .ReturnsAsync((User)null);

        // Act & Assert: verify exception thrown
        await Assert.ThrowsExceptionAsync<Exception>(
            () => _service.GetProfileAsync()
        );
    }
}

ASP.NET Core loads configuration in this priority order (later sources override earlier ones):

1
appsettings.json
Base config. Committed. No secrets. Contains all setting keys with empty values.
2

Service Conventions

  • One interface per service: Every service has a matching IXxxService interface in the Interfaces/ folder

What does this API do?

BDB API is the backend for the BDB (Bouw Database) platform. It provides construction cost data, market indicators, forecasts, and file exports to the BDB frontend and external clients.

Key users: Cost estimators, project managers, BDB data admins, external integration systems.

How do I run the API locally?
  1. Run az login for Key Vault access
  2. Create BdbAPI/appsettings.Development.json with DB connection string
  3. Run dotnet run --project BdbAPI
  4. Open https://localhost:44338/swagger
What is the tech stack?
  • API: ASP.NET Core 8 Web API
  • Database: SQL Server + Entity Framework Core
  • Auth: Azure AD B2C (JWT Bearer)
  • Storage: Azure Blob Storage
  • Excel: Aspose.Cells (licensed)
  • Testing: MSTest + Moq

How do I add a new API endpoint?

Quick steps:

  1. Create interface: Application/Interfaces/IMyNewService.cs
  2. Create service: Application/Services/MyNewService.cs
  3. Register in DI: Application/DependencyInjection.cs — add services.AddScoped<IMyNewService, MyNewService>()
  4. Create/update controller: BdbAPI/Controllers/MyNewController.cs
  5. Write tests: Application.Test/Services/MyNewServiceTest.cs
Where does the business logic go?
ControllersHTTP routing, input extraction, call service, return result. No business logic.
ServicesAll business rules, validations, calculations, orchestration. Core of the app.
RepositoriesDatabase queries only. No business logic, no HTTP awareness.
DomainEntity definitions. No methods beyond simple computed properties.

🔐 Authentication

How does authentication work?

1. Frontend obtains JWT from Azure AD B2C and sends it as Authorization: Bearer {token}

2. API validates JWT using AddMicrosoftIdentityWebApi() (checks signature via JWKS endpoint)

3. Claims are available on HttpContext.User — TokenProvider reads email and client ID

4. Role resolved by AuthorizationService.GetUserRole() from the database

How do I protect an endpoint for admins only?
// Option 1: Throw from service (recommended)
public async Task AdminOnlyAction()
{
    var role = _authService.GetUserRole();
    if (role != "admin")
        throw new ForbiddenAccessException("Admin role required");
    // proceed with admin action...
}

// Option 2: Use AuthUtility helper
if (!AuthUtility.CanAccessByOperator(userRole))
    throw new ForbiddenAccessException("Access denied");

The ForbiddenAccessException is caught by ExceptionMiddleware and returned as HTTP 403.

What is the clientId query parameter?

Every request from the frontend includes ?clientId={companyId}. This identifies which company's data to serve (multi-tenant).

TokenProvider.GetClientId() reads this parameter. Services pass it to repositories for data scoping.

Example: GET /api/ReportForecastFigures/1?clientId=42 — return forecast data for company 42.

🗄️ Database

How do I add a new DB table?
  1. Create entity: Domain/Data/Models/MyEntity.cs implementing IBaseEntity
  2. Add DbSet: Infrastructure/BdbPlatformDbContext.cs — add public DbSet<MyEntity> MyEntity {'{get; set;}'}
  3. Configure model: Add ConfigureMyEntity(modelBuilder) in OnModelCreating()
  4. Add migration: dotnet ef migrations add AddMyEntity
  5. Update DB: dotnet ef database update
Why is my query timing out?
  • Default timeout: 180 seconds (AppSettings:DBConnectionTimeout)
  • Check query plan: Run query in SQL Server Management Studio — look for missing indexes or table scans
  • Use AsNoTracking: Add .AsNoTracking() to read-only queries
  • Projection: Use .Select() to fetch only needed columns
  • Cube queries: DAX queries are especially heavy — consider caching results

📁 File Handling

How does Excel upload work?
  1. Admin sends Excel file as multipart/form-data
  2. Controller calls UploadXxxService.ProcessUploadAsync(file)
  3. Service uses Aspose.Cells to open and parse the workbook
  4. Validates column headers, data types, business rules
  5. On error: throws UploadExcelException → 400 Bad Request
  6. On success: saves entities to DB, stores file in Azure Blob
  7. Optionally: sends email notification via Graph API, logs action
How does Excel report generation work?
  1. Controller calls ReportXxxService.GenerateReportAsync(params)
  2. Service loads the Excel template from App_Data/ (or Azure Blob) using Aspose.Cells
  3. Data from DB/cube is populated into the template cells/ranges
  4. Workbook saved to a MemoryStream
  5. Stream returned as FileContentResult with Content-Disposition: attachment
  6. Frontend browser triggers file download

🐛 Debugging

API returning 401 Unauthorized — how to debug?
  1. Check that Authorization: Bearer {token} header is present in the request
  2. Decode the JWT at jwt.io — check audience (aud), issuer (iss), expiry (exp)
  3. Verify AzureAdB2C:ClientId and Instance match the token's audience and issuer
  4. Check Application Insights for AuthenticationFailed events
API returning 403 Forbidden — why?

Two cases:

  • JwtBearer 403: Token is valid but user doesn't have the required claim/policy
  • ForbiddenAccessException 403: User has a valid token but tried to perform an admin-only action. Check the response body { "error": "..." } for details.
How do I read Application Insights logs?
  1. Go to Azure Portal → Application Insights → BDB API resource
  2. Live metrics: Real-time request/exception feed
  3. Logs (KQL): Query exceptions, requests, traces tables
  4. Failures blade: All 4xx/5xx errors grouped by operation

⚡ Performance

Why are cube/DAX queries slow?
  • DAX queries hit Analysis Services — inherently heavier than SQL queries
  • Avoid triggering cube refreshes during peak hours
  • Consider pre-aggregating results and caching in Redis/in-memory for frequently-requested date ranges
  • The DBConnectionTimeout is set to 180s — if this still times out, the query needs optimization
How to improve EF Core query performance?
  • Always use .AsNoTracking() for read-only queries
  • Use .Select() to project only required columns
  • Avoid N+1 queries — use .Include() or explicit joins
  • Use IQueryable to build queries before calling .ToListAsync()
  • For bulk inserts use AddRange() and a single SaveChanges()

Troubleshooting

Common issues encountered when setting up or developing on the BDB API, and their solutions.

Startup Issues

Key Vault Access Denied

Error: Azure.RequestFailedException — 403 Forbidden from Key Vault
  • You haven't logged into Azure CLI: run az login
  • Your user doesn't have Key Vault Secrets Reader role — ask your team lead to grant access
  • The Key Vault URI in appsettings.Development.json is wrong — verify KeyVaults:BDBKeyVaultName

Database Connection Failed

Solution
  1. Verify ConnectionStrings:DefaultConnection in your local appsettings.Development.json
  2. Check SQL Server is running: sqlcmd -S localhost -Q "SELECT 1"
  3. For LocalDB: sqllocaldb start MSSQLLocalDB
  4. Ensure the database schema is up to date (ask team for latest script or EF migration)

Aspose License Error on Startup

Error: Could not find embedded Aspose.Cells.NET.lic resource

The license file is missing or not set to Embedded Resource in the csproj.

  1. Ensure BdbAPI/License/Aspose.Cells.NET.lic exists (get from team lead)
  2. Verify the csproj has <EmbeddedResource Include="License\Aspose.Cells.NET.lic" />
  3. Rebuild the project: dotnet build BdbAPI

Authentication Issues

401 Unauthorized on All Endpoints

Checklist
  • Is the JWT token being sent in the Authorization: Bearer header?
  • Decode token at jwt.io — is the audience (aud) correct?
  • Is AzureAdB2C:ClientId in appsettings correct?
  • Is the token expired? Check exp claim
  • Is the B2C policy correct? B2C_1A_BDB_SIGNUP_SIGNIN

403 Forbidden on Admin Endpoints

// Check the response body for the error message
// e.g. { "error": "Admin role required" }

// Verify your user has admin role in the DB
// Table: UserRole — check RoleId for your UserId and the clientId you're using

Build Issues

NuGet Restore Fails

Solution
  1. Check internet / VPN connectivity to nuget.org
  2. Clear local NuGet cache: dotnet nuget locals all --clear
  3. Re-run: dotnet restore BDB-API.sln
  4. If a private package fails, verify Azure Artifacts/private NuGet source credentials

Build Errors: Nullable Reference Warnings

Solution
  • Nullable is enabled — handle null cases explicitly
  • Use null-forgiving operator ! only as last resort: user!.Email
  • Prefer null checks: if (user == null) throw new ...

Test Issues

Tests Failing Due to Missing Moq Setup

// Error: Invocation was not matched in Mock setup
// Fix: Add a Setup() for the called method
_mockRepo.Setup(r => r.GetByEmailAsync(It.IsAny<string>()))
    .ReturnsAsync(new User { ... });

// Or use It.IsAny<T>() to match any parameter value
// Or call mock.SetupAllProperties() for property getters

Tests Pass Locally But Fail in CI

Common Causes
  • DateTime.Now dependency: Use injectable IDateTimeProvider instead of DateTime.Now in services
  • File path differences: Use Path.Combine() instead of hardcoded separators
  • Parallel test execution: If tests share static state, they fail when run in parallel
  • Missing test data: Test setup ([TestInitialize]) must create all required data

Excel Issues

Aspose Running in Evaluation Mode (Watermarks)

Problem: Excel files have "Evaluation" watermarks

Aspose license not loaded correctly.

  1. Check Startup.SetAsposeLicense() is called before any Aspose operations
  2. Verify the license file resource name matches: "BdbAPI.License.Aspose.Cells.NET.lic"
  3. Ensure the license file hasn't expired — contact Arcadis procurement for renewal

Excel Template Not Found

// Error: File not found at path App_Data/ExcelTemplates/...
// Cause: App_Data/ not included in publish output

// Fix: Ensure App_Data is set to "Copy to output directory" in the csproj
<ItemGroup>
  <None Update="App_Data/**">
    <CopyToOutputDirectory>PreserveNewest</CopyToOutputDirectory>
  </None>
</ItemGroup>

Performance Issues

Request Timeouts on Large Datasets

  • Increase AppSettings:DBConnectionTimeout in config (default 180s)
  • Add indexes on frequently-filtered columns in SQL Server
  • Consider moving heavy operations to the background task queue and returning 202 Accepted
  • Use stored procedures for complex joins instead of EF LINQ for better query plan control
