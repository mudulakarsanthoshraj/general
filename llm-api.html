The Ask & Cost

Enable controlled access to one low-cost Large Language Model API (recommended: GPT-4o Mini or an approved alternative) so teams can build, measure, and learn from real AI use cases. Today’s network restrictions make even a small, safe pilot hard. The Luma AI Assistant in this document is the first use case.

Typical spend is ~$1–$30 per use case per month under a $50/month hard cap. Token rates, model catalogue, and cost comparison are in OpenAI Model Pricing.

Beyond the Luma chatbot

The same controlled API key powers more than the Luma chatbot. Each future vetted use case rides the same key, the same $50/month cap, and the same guardrails — so the marginal cost of the next approved use case is near zero. Whether it pays off is measured per use case (see How We’ll Measure the Savings in the Luma proposal); the Luma chatbot is the first such measurement.

Scope of this Ask

What we’re asking management to approve — and what we are explicitly not asking for. Network policy (Zscaler) and existing security controls still apply.

To avoid any scope confusion: this is a tightly bounded enablement — one approved outbound API endpoint, used server-side by the Luma AI Assistant chatbot. It is not a request to lift network policy, and it is separate from any conversation about personal AI tooling (Copilot, Claude, ChatGPT).

What this proposal asks for

  • Outbound access to one approved API endpoint — server-side; Luma chatbot first.
  • A Zscaler exception via the standard Security exception process for that one URL.
  • A reusable pattern so future vetted use cases can share the same enablement (operational guardrails in How It Stays Safe).

What this proposal does not ask for

  • Per-engineer or per-laptop API keys.
  • GitHub Copilot, Claude desktop, or ChatGPT in the browser — separate decisions with different trade-offs.
  • A blanket lift of Zscaler or any company-wide network policy.
  • Access to multiple vendors at once — one vendor, one URL, reviewed.
On Zscaler / network policy
We are not bypassing Zscaler. We use the same exception process as other approved SaaS APIs to allow one outbound URL for one reviewed use case. Network policy continues to apply everywhere else. Personal AI tooling remains a separate conversation — this proposal does not pre-decide it.

What this means for the Luma chatbot

A typical Luma chatbot question costs about 2,000 input tokens (system prompt + retrieved Luma docs + the user question) and 600 output tokens (the answer). Applying the catalogue above to a moderate-usage scenario of 10,000 questions per month:

Cost = (2,000 input tokens × input rate + 600 output tokens × output rate) ÷ 1,000,000. Rounded to two significant figures.

Why GPT-4o mini
For documentation Q&A, GPT-4o mini delivers flagship-class answer quality at ~17× lower cost than GPT-4o and ~100× lower than the reasoning models. Even cheaper nano-class models exist, but answer quality drops on multi-clause questions like “outline button with themeColor="error" in Angular with a Tabler icon”. GPT-4o mini hits the sweet spot — strong answers, predictable spend, comfortably inside the $50/month cap.

Ollama — Free Local Alternative

Ollama is a free, open-source way to run Large Language Models on your own machine. We already use it for development — this section explains why it can’t be the production answer for Luma, and how the two fit together.

ollama.com runs open models (Llama 3, Mistral, Phi, etc.) on a developer’s machine — no API key, no per-question cost, data never leaves the laptop. The Luma prototype already supports Ollama and OpenAI-compatible APIs for local prompt work without API spend.

Why Ollama can’t be the production deployment

What Ollama is great for

  • Prompt iteration and offline exploration on your own hardware
  • Useful when outbound API access is blocked — entirely local, no Zscaler exception needed

Why it falls short for production

  • Every Luma consumer would need to install Ollama + download a ~2–8 GB model — impractical
  • Performance depends on each user’s machine (CPU/GPU/RAM); answers can take 10–30 seconds on a typical laptop
  • Smaller open models give noticeably lower answer quality than GPT-4o Mini for the same prompt
  • Hosting Ollama centrally on a shared server brings back ops + GPU costs that far exceed the $50/month API budget
  • No vendor SLA, no audit tooling, no managed updates
How both fit together
Use Ollama free for local development and prompt iteration. Use the low-cost GPT-4o Mini API for the production chatbot served inside Luma Storybook. The codebase already supports both — switching providers is a one-line URL change, so we keep the freedom to revisit later as open models improve.

How It Stays Safe

Guardrails baked into every use case: one approved model, capped spend, no sensitive data, sign-off per use case.

Guardrail How it works
One approved model Start with GPT-4o Mini only. New providers go through Security review.
Hard budget cap Pay-as-you-go with a monthly ceiling per use case (for example, $50/month for the Luma chatbot). Automatic shut-off if exceeded.
No sensitive data Public documentation and synthetic data only — no personal data (names, emails, employee IDs), no proprietary code, no credentials.
Sign-off per use case One-page intent doc (owner, data classification, spend cap, success metric) signed off by Security, Platform, and Luma before going live.

At a glance

Luma is Arcadis Digital Intelligence’s shared design system, built on Kendo UI — components and design tokens for React, Angular, and Web Components teams. This proposal upgrades Luma’s existing Storybook assistant into a low-cost, AI-grounded chatbot that answers developer questions from our own documentation.

  • Problem: ~4 hours per week lost to repeat developer questions on Slack — answers already exist in Storybook.
  • Solution: Upgrade the Storybook assistant to an AI chatbot grounded in Luma docs (GPT-4o Mini or equivalent).
  • Cost: ~$1–$30 per month in API usage; hard cap $50 per month.
  • Value: ~14 hours per month of engineering time recovered (~$600 per month).
  • Timeline: 2 months — pilot month 1, open rollout month 2 if successful.
  • Ask: Approve LLM API access + $50/month cap + a small review group.

Why We Need This

Luma adoption is growing faster than the design system team can answer questions manually.

Multiple Arcadis product teams now consume Luma across React, Angular, and Web Component stacks. As adoption grows, the design system team spends an ever-larger share of each day on the same repeated questions from those teams.

Today, on Slack / Teams / DMs “How do I use the Button component?” · “How many variants does it have?” · “How do I install Luma in Angular?” · “Which themeColor values are supported?” · “Where are the design tokens documented?”

These questions are already answered in our Storybook docs, but consuming teams don’t find them, don’t know where to look, or find it faster to ping a developer. The result:

  • Design system engineers lose about 4 hours every week to repeat conversations
  • Consuming teams wait minutes or hours for answers that exist in our docs
  • Each new adopting team multiplies the support load — support time grows linearly with adoption
  • Off-hours and cross-region questions wait until the next working day

A static Frequently Asked Questions list isn’t enough — real developer questions come in natural language, with many variations and combinations (for example, “an outline button with themeColor="error" in Angular with a Tabler icon”).

Proposed Solution

An AI Assistant Chatbot inside Luma Storybook, powered by a low-cost LLM API and grounded in our own documentation.

Luma Storybook already ships a basic keyword-matched assistant. We propose upgrading it to an AI Assistant Chatbot powered by a small, low-cost Large Language Model API — GPT-4o Mini, Claude Haiku, or Gemini Flash. Same user interface, smarter answers.

The chatbot is grounded in our actual documentation:

  • Luma component stories and argTypes (the source of truth for props)
  • Foundation docs (design tokens, theming, accessibility)
  • Setup guides (React, Angular, Web Components, Tabler icons)
  • Storybook MDX pages

The model only sees these docs — no customer data, no business logic, no product source. It returns a clean, conversational answer with code samples.

Who builds it: The Luma team. The chatbot user interface already exists in Storybook; this is a focused upgrade (swap the keyword matcher for a model call), not a new product.

Effort to ship: A working prototype already runs in our local Storybook (grounded retrieval over 174 Luma documentation chunks, Ollama and OpenAI both wired in). Remaining work to pilot: about 1 week of one engineer — audit logging, the personal-data filter, and the review dashboard.

(recommended) 		Upgrade the existing chatbot to call GPT-4o Mini (or equivalent), grounded in Luma documentation. Lowest cost (~$1–$30 per month), fastest to ship (~1 week), reuses the existing user interface, scales by usage not headcount.

How It Helps

Five direct gains for Luma, the design system team, and every product team consuming Luma.

  • Faster Luma adoption. Consuming teams get unblocked instantly instead of waiting on a person. Lower friction = more teams adopting the design system = more consistent user experience across products.
  • Better developer experience. Asking the chatbot is faster than scrolling docs or pinging Slack — answers in seconds.
  • Documentation that explains itself. New components, tokens, or guidelines added to Luma are picked up automatically. No separate Frequently Asked Questions list to maintain.
  • Consistency across frameworks. The assistant answers React, Angular, and Web Component questions equally — hard for any one human to do consistently.
  • A foundation for AI inside our products. Once access, cost controls, and security guardrails are proven, the same pattern powers AI features in customer-facing products.

How We’ll Measure the Savings

The savings above are a directional estimate. This section is how we prove them — or stop, cleanly, if they don’t hold up. Baseline first, then weekly tracking, then an explicit go / no-go review at week 4.

The four phases

Phase When What we measure
1. Baseline Week 0 — before pilot starts Luma-related questions per week (Slack / Teams / DMs); Luma team self-reported support time; topic mix (component, install, theming, …).
2. Pilot tracking Weeks 1 – 4 Same questions count (has it dropped?); chatbot usage; weekly accuracy spot-check on a sample of 20 answers; API spend; hours-saved estimate.
3. Decision gate End of week 4 Explicit go / no-go against the thresholds below. If thresholds aren’t met, the chatbot is turned off and the API key is revoked.
4. Open-rollout tracking Weeks 5 – 8 Same metrics across all Luma developers; confirm the pattern scales; final hours-saved and ROI reported to management at week 8.

Decision gate — end of week 4

Concrete, written-down thresholds. We continue only if all three are met:

Hours saved / month
≥ 8 hrs
Answer accuracy
≥ 70%
API spend
≤ $50 cap

“Hours saved” = drop in Luma-team support time vs the week-0 baseline. “Accuracy” = % of sampled answers the Luma team rates as correct and grounded in our docs (random 20-answer sample, weekly).

If the numbers don’t hold up
The chatbot is switched off at the end of week 4, the API key is revoked, and the proposal is closed. Sunk cost is capped at ~$50 in API spend over the pilot — no infrastructure, no licensing, no headcount commitments. This decision gate is in writing so it’s real, not aspirational.

What gets reported to management

  • Weekly during pilot: one-line status — questions count, chatbot usage, accuracy sample, spend-to-date.
  • End of week 4: written go / no-go memo against the three thresholds above.
  • End of week 8: final ROI report — baseline vs measured hours-saved, actual API spend, and recommendation on next use cases.

Eight controls that bound the spend, the data, and the answer quality.

The Ask

Three decisions we’re asking management to approve.

  1. Approve controlled outbound access to one low-cost Large Language Model API (recommended: GPT-4o Mini, or approved equivalent) for the Luma Design System.
  2. Approve an ongoing $50/month budget cap on the API key.
    Expected actual spend: ~$1–$30 per month. The cap is a safety ceiling, not a flat bill — pay-as-you-go, with automatic shut-off if reached.
  3. Assign a small reviewer group (one each from Security, Platform, and the Luma team) to sign off on guardrails and oversee the pilot.
Bottom line
For a downside capped at $50 per month, we eliminate a recurring drain on engineering time, accelerate Luma adoption, and establish a safe, measured pattern for broader AI adoption.
