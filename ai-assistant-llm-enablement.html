Proposal

AI Assistant Chatbot for the Luma Design System

Controlled, low-cost Large Language Model (LLM) API access to cut repeated support work and scale Luma adoption.
Context: Luma is Arcadis Digital Intelligence’s shared design system, built on Kendo UI — components and design tokens for React, Angular, and Web Components teams. This proposal upgrades Luma’s existing Storybook assistant into a low-cost, AI-grounded chatbot that answers developer questions from our own documentation.

At a glance

  • Problem: ~4 hours per week lost to repeat developer questions on Slack — answers already exist in Storybook.
  • Solution: Upgrade the Storybook assistant to an AI chatbot grounded in Luma docs (GPT-4o Mini or equivalent).
  • Cost: ~$1–$30 per month in API usage; hard cap $50 per month.
  • Value: ~14 hours per month of engineering time recovered (~$600 per month).
  • Timeline: 2 months — pilot month 1, open rollout month 2 if successful.
  • Ask: Approve LLM API access + $50/month cap + a small review group.

01 Why We Need This

Multiple Arcadis product teams now consume Luma across React, Angular, and Web Component stacks. As adoption grows, the design system team spends an ever-larger share of each day on the same repeated questions from those teams.

Today, on Slack / Teams / DMs “How do I use the Button component?” · “How many variants does it have?” · “How do I install Luma in Angular?” · “Which themeColor values are supported?” · “Where are the design tokens documented?”

These questions are already answered in our Storybook docs, but consuming teams don’t find them, don’t know where to look, or find it faster to ping a developer. The result:

  • Design system engineers lose about 4 hours every week to repeat conversations
  • Consuming teams wait minutes or hours for answers that exist in our docs
  • Each new adopting team multiplies the support load — support time grows linearly with adoption
  • Off-hours and cross-region questions wait until the next working day

A static Frequently Asked Questions list isn’t enough — real developer questions come in natural language, with many variations and combinations (for example, “an outline button with themeColor="error" in Angular with a Tabler icon”).

02 Proposed Solution: An AI Assistant Chatbot in Luma

Luma Storybook already ships a basic keyword-matched assistant. We propose upgrading it to an AI Assistant Chatbot powered by a small, low-cost Large Language Model API — GPT-4o Mini, Claude Haiku, or Gemini Flash. Same user interface, smarter answers.

The chatbot is grounded in our actual documentation:

  • Luma component stories and argTypes (the source of truth for props)
  • Foundation docs (design tokens, theming, accessibility)
  • Setup guides (React, Angular, Web Components, Tabler icons)
  • Storybook MDX pages

The model only sees these docs — no customer data, no business logic, no product source. It returns a clean, conversational answer with code samples.

Who builds it: The Luma team. The chatbot user interface already exists in Storybook; this is a focused upgrade (swap the keyword matcher for a model call), not a new product.

Effort to ship: A working prototype already runs in our local Storybook (grounded retrieval over 174 Luma documentation chunks, Ollama and OpenAI both wired in). Remaining work to pilot: about 1 week of one engineer — audit logging, the personal-data filter, and the review dashboard.

Today (Manual Support)

  • Same questions asked repeatedly
  • Developer interrupted multiple times/day
  • Answers arrive in minutes or hours
  • Off-hours = no answer
  • Scales linearly with adoption

With AI Assistant Chatbot

  • Instant answers, 24 / 7
  • Design system team protected from repeated pings
  • Same quality for 10 or 10,000 users
  • Stays in sync with docs automatically
  • Cost scales by usage, not by team size

03 Alternatives Considered

Before recommending the AI-grounded chatbot, we evaluated four alternatives — each weighed on cost, time to deliver, and ability to scale with Luma adoption.

Option What it is Why it falls short
A. Improve documentation only Continue investing in Storybook docs, tutorials, examples. Already in place. Doesn’t change the support load — consuming teams keep asking the same questions instead of searching the docs.
B. Add a dedicated support engineer Hire one engineer focused on Luma developer support. Roughly $100K+ per year fully loaded. Support load scales linearly with adoption — headcount doesn’t.
C. Static Frequently Asked Questions chatbot The keyword-matched assistant we already have in Storybook. Can’t handle natural-language variations or combinations. Needs constant manual upkeep.
D. Commercial chatbot product Procure a vendor SaaS chatbot (Intercom, Drift, etc.). Typically $10K–$50K per year in licensing. Vendor lock-in, generic answers, and no native grounding on Luma-specific Storybook docs.
E. AI Assistant + low-cost LLM API
(recommended) 		Upgrade the existing chatbot to call GPT-4o Mini (or equivalent), grounded in Luma documentation. Lowest cost (~$1–$30 per month), fastest to ship (~1 week), reuses the existing user interface, scales by usage not headcount.

04 How This Helps Luma & Consuming Products

  • Faster Luma adoption. Consuming teams get unblocked instantly instead of waiting on a person. Lower friction = more teams adopting the design system = more consistent user experience across products.
  • Better developer experience. Asking the chatbot is faster than scrolling docs or pinging Slack — answers in seconds.
  • Documentation that explains itself. New components, tokens, or guidelines added to Luma are picked up automatically. No separate Frequently Asked Questions list to maintain.
  • Consistency across frameworks. The assistant answers React, Angular, and Web Component questions equally — hard for any one human to do consistently.
  • A foundation for AI inside our products. Once access, cost controls, and security guardrails are proven, the same pattern powers AI features in customer-facing products.

07 Security & Cost Controls

Control How it works
Hard monthly budget cap $50 ceiling on the API key; automatic shut-off when reached.
Data sent to the model Only public Luma docs and the user’s question. No customer data, no product source.
No training on our prompts Vendor’s “zero data retention” / no-training tier.
Approved vendor only Single vendor, terms reviewed by Security & Legal.
Audit trail All prompts and responses logged 30 days for security review.
Personal data filter (names, emails, employee IDs, etc.) Prompts scanned and blocked when they match known sensitive patterns.
Answers grounded in Luma docs only Retrieval-Augmented Generation — the model is fed relevant Storybook snippets before each answer, so it can’t invent components or properties.
Unknown questions & quality review Clear “I don’t know” fallback when no doc matches; weekly answer sampling by the Luma team; user-reportable answers.

08 Rollout Plan — 2 Months

Full rollout fits inside 2 months, in two clear steps. Each step depends on the previous one — we can stop at any point if the results don’t justify continuing.

Step When What happens How we judge success
1. Pilot
(internal only)		 Weeks 1–4 Enable the AI Assistant Chatbot inside Luma Storybook for the Luma team and 1–2 invited product teams. They use it for real Luma questions instead of pinging Slack. • Does it answer correctly?
• Are people actually using it?
• Does Slack traffic to the Luma team drop?
2. Open Rollout
(all Luma users)		 Weeks 5–8 If the pilot works, open the chatbot to every Luma developer across the organization. No new infrastructure — same chatbot, broader audience. • Hours saved per Luma maintainer
• Satisfaction from consuming teams
• Cost stays inside the $50 cap
Success targets (by end of month 1)
  • At least 70% of common Luma questions answered without a human
  • Measurable drop in repeat Slack / Teams questions to the Luma team
  • Monthly API spend stays under the $50 cap
In one line Month 1: prove it works with a small group inside Luma. Month 2: if it works, open it to every Luma developer.
