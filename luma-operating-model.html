Luma Design System Operating Model

How Luma can evolve to support AI-assisted design system adoption, implementation, and maintenance across multiple frontend stacks.

This document was written with the help of AI. The concepts, architecture, and workflows described here should be viewed as a working model rather than a fixed blueprint. Capabilities of AI tools, design tooling, and development ecosystems are evolving rapidly — many approaches will need validation through experimentation. Principles and objectives are expected to remain stable; implementation details, tooling choices, workflows, and levels of automation may change over time.

Purpose

This document describes the architecture, operating model, and artefacts that make up the Luma Design System ecosystem.

It explains:

  • How design intent is authored and maintained
  • How design information is structured
  • How design specifications are generated
  • How AI agents consume those specifications
  • How implementation is generated across different frontend stacks
  • How design system assets are governed and maintained
Scope of this document
This document focuses on the overall system architecture and flow of information. Detailed authoring instructions for designers and implementation guidance for engineers are covered in separate documents.

System Overview

Luma consists of four layers, each with a specific responsibility and ownership model. The purpose is to separate design intent from implementation, allowing the same design system to be applied consistently across multiple frontend frameworks.

Foundations

Visual and interaction rules that underpin the system, represented through design tokens.

  • Color
  • Typography
  • Iconography
  • Spacing
  • Elevation
  • Motion
  • Accessibility
  • Illustrations

Components

Reusable UI elements with a specific responsibility. Defined independently of any frontend framework or component library — implementation details are introduced later through framework-specific adaptation.

  • Button
  • Input
  • Select
  • Checkbox
  • Card
  • Badge
  • Table

Patterns

Reusable solutions to recurring user problems, composed of multiple components. Patterns provide the majority of consistency and efficiency benefits.

  • Page Header
  • Side Navigation
  • Filter Panel
  • Form
  • Data Table
  • Empty State
  • Action Bar

Templates

Page-level compositions built from multiple patterns — proven starting points for common product experiences.

  • Dashboard
  • List View
  • Detail View
  • Create/Edit Flow
  • Wizard
  • Reporting View

Authoring Layer

Where design intent is created and maintained. Figma serves as the primary source of truth for design intent.

Primary tooling
  • Figma — design authoring environment
  • Luma Design Library — shared component and pattern library

Designers are responsible for defining not only how something looks, but also how it behaves, how it should be used, and how it should be composed with other elements. Design artefacts should be structured so they can be interpreted consistently by designers, engineers, and AI agents.

Designing for Adaptation

Designers should define solutions based on user needs and Luma standards while remaining aware of the capabilities and constraints of supported implementation frameworks.

General principle
  • Reuse existing capabilities where they meet the user need.
  • Extend existing capabilities only when necessary and reasonable.
  • Create new components or patterns only when existing solutions are insufficient and the value outweighs the investment.

Design decisions should consider:

  • User value
  • Accessibility
  • Consistency
  • Implementation complexity
  • Maintainability
  • Long-term ownership

Avoid introducing custom components, wrappers, or framework-specific extensions unless they provide clear value that cannot be achieved through existing capabilities. Where trade-offs exist, designers should collaborate with engineers before introducing new patterns or component requirements.

Specification Layer

Structured, framework-agnostic representation of design intent — consumable by both humans and AI agents. This structure is a directional guide, not a fixed blueprint.

Rather than generating implementation directly from Figma, design intent is documented through structured artefacts that describe the foundations, components, patterns, and templates that make up Luma.

tokens.json

Visual foundations: colors, typography, spacing, motion, elevation, border radius. Source of truth for all implementations.

design-system.md

Global rules: design principles, naming conventions, accessibility, interaction conventions, theming, framework strategy.

component.md

Component contracts: purpose, anatomy, variants, states, behavior, accessibility, token mappings, usage guidance.

pattern.md

Interaction patterns: structure, hierarchy, composition rules, responsive behavior, accessibility, usage guidance.

template.md

Page-level templates: layout regions, pattern composition, responsive behavior, page-level accessibility, usage guidance.

Specification Generation

Artefacts may be authored manually, generated through AI-assisted workflows, or maintained through a combination of both. Designers remain responsible for defining design intent; AI agents assist in creating, maintaining, enriching, and updating specifications.

  1. A designer creates or updates a component, pattern, or template in Figma.
  2. An AI agent assists in generating or updating the corresponding specification.
  3. The designer reviews, refines, and approves the generated output.
  4. The approved specification becomes part of the Luma source of truth.
Primary objective
Not the generation process itself — but ensuring design intent is represented in a structured, reusable, and machine-readable format that can be interpreted consistently across teams, tools, AI agents, and frontend frameworks. The specification artefacts remain the canonical representation of design intent as tooling evolves.

Adaptation Layer

Converts structured design specifications into framework-specific implementation proposals. Framework mappings are an input into this layer, not a separate layer.

The same Luma Form pattern may result in an Angular + Kendo implementation, a React + Material UI implementation, or a React + Shadcn implementation — depending on the target product stack. The design intent remains constant; only the implementation strategy changes.

AI agent responsibilities

  • Reading design specifications and understanding design intent
  • Reading framework mappings
  • Translating specifications into framework-specific implementations
  • Generating implementation proposals and pull requests

Examples

  • Translating Luma design tokens into CSS variables, Sass themes, or framework theme configs
  • Translating a Luma Button spec into Kendo, Material UI, or Shadcn
  • Translating a Form pattern into a framework-specific composition
  • Translating a Dashboard template into a page aligned with Luma standards
  • Translating existing applications to align with updated Luma standards

Key principle
Luma supports multiple frontend stacks without duplicating design system definitions. One specification, many implementations.

Delivery Layer

Packaging, publishing, and distributing Luma assets for consumption by product teams.

Stage 1 — Package-Based Adoption

Objective

Provide product teams with production-ready Luma assets and implementation packages consumable within their existing frontend stacks.

Luma publishes versioned packages through the Luma repository and package registries:

  • @luma/tokens
  • @luma/icons
  • @luma/angular-kendo
  • @luma/react-material
  • @luma/react-shadcn

Published assets may include design tokens, icon library, font packages, component packages, pattern implementations, template implementations, and documentation. Product teams consume and adopt these according to their technology stack and release cadence.

Stage 2 — AI-Assisted Adoption

Objective

Reduce the effort required for teams to understand, adopt, and apply Luma standards.

AI is introduced as an engineering assistant, not an automation engine. Engineers continue to own implementation decisions; AI helps identify opportunities, answer questions, and accelerate adoption.

AI agents may assist with:

  • Analyzing existing codebases
  • Identifying opportunities to adopt Luma components, patterns, and tokens
  • Recommending replacements for custom implementations
  • Assisting with upgrades to newer Luma versions
  • Generating implementation proposals and pull requests
  • Helping migrate applications toward Luma standards incrementally
Example
An AI agent analyzes an existing application and identifies components that can be replaced by Luma equivalents, suggests token migrations, or generates implementation proposals that align existing screens with established Luma patterns and templates. Engineers remain responsible for validation, testing, and approval.

Ownership Model

Clear ownership across the design system team, product teams, and AI agents.

Role Responsible for Does not own
Design System Team Foundations, components, patterns, templates, design specifications, governance, design system evolution —
Product Teams Product implementation, product-specific requirements, contributions back into Luma, adoption and rollout —
AI Agents Translating specifications into framework-specific implementations, applying tokens and theming rules, mapping components to framework equivalents, generating implementation proposals and pull requests, assisting with upgrades and migrations Design decisions, product decisions, design system governance, quality approval
Human judgment stays central
AI agents accelerate work and reduce manual effort, but design decisions, product decisions, governance, and quality approval always remain with people. Engineers validate, test, and approve every AI-generated proposal before it ships.
