At a glance

How to run the Luma AI Assistant chatbot on your own machine with Ollama — free, offline, and no API key. About 10 minutes end to end.

Cost
$0
Setup time
~10 min
Model size
~2–8 GB
Internet
Only to install

What you’ll do

  1. Install Ollama (one-click installer for macOS, Windows, Linux).
  2. Pull a model — llama3.2 is the default and works well.
  3. Start the Ollama server with one extra environment variable so the browser can call it.
  4. Point the Luma chatbot at Ollama by editing .env.
  5. Run Storybook — you’re chatting with a local model.
Why this matters
Ollama lets you iterate on the chatbot — prompts, retrieval, UI — without burning any API spend, and without needing a Security exception for outbound network access. Great for local development and demos. For production, the project uses an approved hosted model (see the main proposal).

What You’ll Need

A reasonably modern laptop and the Luma repo checked out. Nothing else.

A note on performance
Local models run on your CPU (or GPU if you have one). Expect first-token latency of a few seconds on a typical laptop — not as snappy as a hosted API, but plenty fast for development.
1 Install Ollama

Install Ollama

One installer for your platform. Takes about a minute.

macOS & Windows

Download the installer from ollama.com/download and run it. On macOS this installs the ollama menu bar app and CLI; on Windows it installs the service and CLI.

Linux (one-line install)

Terminal Linux 
curl -fsSL https://ollama.com/install.sh | sh

Verify the install

Open a new terminal and run:

Terminal All platforms 
ollama --version

You should see a version like ollama version 0.x.x. If the command is not found, restart your terminal so the new PATH is picked up.

Done
Ollama is installed. Move to step 2.
2 Pull a Model

Pull a Model

Download llama3.2 once — the chatbot is wired to use it by default. About 2 GB.

Terminal One-time 
ollama pull llama3.2

You’ll see a progress bar while the model downloads. Once done, the model lives on your disk — you only need to do this once.

Verify the model is available

Terminal Optional 
ollama list

You should see llama3.2 in the list with a size around 2 GB.

Want a different model?
See Choosing a Model for alternatives like qwen2.5:7b or mistral. Pull whichever you prefer — you can switch later by editing one line in .env.
3 Start the Server

Start the Ollama Server

Start Ollama with one extra environment variable so the Storybook browser tab can call it. This is the step most people get wrong.

Why OLLAMA_ORIGINS matters
The chatbot runs in your browser at localhost:6006. By default, Ollama only accepts requests from localhost:11434 — the browser will see a CORS error and the chatbot will fail to connect. Setting OLLAMA_ORIGINS="*" tells Ollama to accept browser requests from any local origin.

macOS & Linux

Terminal — leave this running Every session 
OLLAMA_ORIGINS="*" ollama serve

Windows (PowerShell)

PowerShell — leave this running Every session 
$env:OLLAMA_ORIGINS="*"; ollama serve

You should see something like Listening on 127.0.0.1:11434 in the terminal. Leave this terminal open — it’s the model server. If you close it, the chatbot stops working.

If Ollama is already running

On macOS the menu bar app may auto-start Ollama without the right OLLAMA_ORIGINS. Quit it from the menu bar first, then start it from the terminal with the command above.

Terminal — stop any background Ollama first macOS 
# Quit Ollama from the menu bar, OR:
pkill ollama
# Then start it correctly:
OLLAMA_ORIGINS="*" ollama serve
Verify it’s up
In a new terminal: curl http://localhost:11434/api/tags. You should get a JSON list of installed models — including llama3.2.
4 Configure the Chatbot

Point the Chatbot at Ollama

Edit one file — .env in the Luma repo root — so the chatbot talks to Ollama instead of OpenAI.

If you don’t have a .env file yet, copy the example:

Terminal — from the repo root One-time 
cp .env.example .env

Set these three lines

Open .env and make sure it contains exactly:

.env Ollama 
# Ollama (local, free) — no API key required
STORYBOOK_LLM_API_KEY=
STORYBOOK_LLM_API_URL=http://localhost:11434/v1/chat/completions
STORYBOOK_LLM_MODEL=llama3.2

That’s it. Three lines:

  • API key — left empty. Ollama doesn’t use one and will actually complain if you send a bogus header.
  • API URL — the OpenAI-compatible endpoint Ollama serves on port 11434.
  • Model — must match a model you’ve pulled (ollama list).
Restart Storybook after editing .env
Storybook reads environment variables once at boot. If Storybook was already running, stop it (Ctrl+C) and start it again so the new values take effect.
5 Run Storybook

Run Storybook & Chat

With Ollama serving and .env set, start Storybook and open the assistant.

Terminal — from the repo root Run 
npm run storybook

Storybook builds the Luma documentation corpus, then opens http://localhost:6006. The AI Assistant button is in the bottom-right corner.

Confirm you’re using Ollama

Open the assistant. The status banner should read “Connected to Ollama (local, free) at http://localhost:11434/v1/chat/completions”. If you see “OpenAI requires an API key” instead, your .env didn’t take effect — restart Storybook.

You’re done
Ask a real Luma question — for example, “How do I use the outline Button with a Tabler icon in Angular?” The first answer will take a few seconds (model warming up). Subsequent answers are faster.

Next time you want to use it

Two terminals. That’s the whole loop:

Terminal 1 — model server Keep open 
OLLAMA_ORIGINS="*" ollama serve
Terminal 2 — Storybook Keep open 
npm run storybook

Switching Back to OpenAI

One file change. Useful for testing the production model or comparing answer quality side by side.

.env OpenAI 
# OpenAI (paid, cloud)
STORYBOOK_LLM_API_KEY=sk-your-real-key-here
STORYBOOK_LLM_API_URL=https://api.openai.com/v1/chat/completions
STORYBOOK_LLM_MODEL=gpt-4o-mini

Restart Storybook. The status banner should now say “Connected to OpenAI”.

Ollama (local)

  • Free; no API key
  • Data stays on your laptop
  • Slower first response; smaller model = lower answer quality
  • Great for prompt iteration and offline work

OpenAI (production)

  • Paid — pay-as-you-go (~$0.0007 / question for GPT-4o Mini)
  • Requires Security-approved API key
  • Faster, sharper answers
  • What we ship in production

Choosing a Model

llama3.2 is the default and a good starting point. Here are reasonable alternatives if you want sharper answers or have a beefier machine.

Model Size on disk Best for Pull command
llama3.2 (default) ~2 GB Lightweight, runs comfortably on 8 GB laptops ollama pull llama3.2
qwen2.5:7b ~4.7 GB Stronger reasoning; good code answers ollama pull qwen2.5:7b
mistral ~4.1 GB Well-rounded general-purpose ollama pull mistral
llama3.1:8b ~4.7 GB Higher quality than 3.2, more RAM needed ollama pull llama3.1:8b
phi3 ~2.3 GB Microsoft Phi-3; compact alternative to llama3.2 ollama pull phi3

To use a different model: pull it, then change STORYBOOK_LLM_MODEL in .env to the model name (exactly as it appears in ollama list), and restart Storybook.

Models are listed at ollama.com/library. Sizes are approximate and may vary by quantisation.

Troubleshooting

The five issues that account for almost every Ollama setup problem.

The chatbot says “Could not reach Ollama”

  • Ollama isn’t running. Check Terminal 1 — the ollama serve process needs to stay open.
  • Started without OLLAMA_ORIGINS. Quit Ollama (menu bar on macOS, pkill ollama elsewhere) and restart with OLLAMA_ORIGINS="*" ollama serve. This is the #1 cause.
  • Port 11434 in use. Another process owns the port. Stop it, or change Ollama’s port and update STORYBOOK_LLM_API_URL.

CORS error in the browser console

Same root cause as above — OLLAMA_ORIGINS wasn’t set when Ollama started. Restart Ollama with the env var.

“Model llama3.2 not found”

You didn’t pull the model, or STORYBOOK_LLM_MODEL doesn’t match what ollama list shows. Common mismatches: llama3 vs llama3.2, missing the :7b suffix on Qwen.

Status banner still says “OpenAI requires an API key”

Storybook is reading old environment variables. Stop Storybook (Ctrl+C) and run npm run storybook again. The .env file is only read at boot.

First answer is very slow

Normal — the model is loading into RAM on the first request. Subsequent answers are much faster (model stays warm). If it stays slow, the model is bigger than your RAM comfortably handles; try a smaller model (see Choosing a Model).

Still stuck?
Check the browser DevTools console (Network tab) for the failing request to localhost:11434. Status 0 / red almost always means OLLAMA_ORIGINS. Status 404 usually means the model name is wrong. Status 200 with no answer means the model returned an empty string — try a different model.
