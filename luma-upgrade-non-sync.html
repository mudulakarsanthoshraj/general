Executive Summary

This document records every meaningful gap between the proposed Stage 1 (Package-Based Adoption) and Stage 2 (AI-Assisted Adoption) upgrade vision and what the luma-design-system repository actually delivers today.

Bottom line
The upgrade proposal describes a multi-package, multi-framework design system platform with AI-driven codebase migration. The current repo is a single private npm package that ships one compiled CSS file for Kendo UI only, with Storybook as documentation and a local keyword chatbot — not a repo-aware AI agent.

What exists today

  • @arcadis-intelligence/luma-design-system → dist/luma-theme.css
  • Tokens, fonts, icons bundled inside compiled CSS
  • Kendo React/Angular usage documented in Storybook
  • Page Shell patterns as copy-paste templates only
  • LumaAssistant — Storybook FAQ chatbot

What the proposal assumes

  • Granular packages: @luma/tokens, @luma/icons, @luma/angular-kendo, @luma/react-material, @luma/react-shadcn
  • Versioned component & pattern npm packages
  • Multi-framework adoption paths
  • AI agents that read product repos and generate PRs
In one line
Stage 1 is not an incremental upgrade — it is a replatform. Stage 2 is not an extension of LumaAssistant — it is a separate engineering platform that depends on Stage 1 having stable package APIs and a component mapping catalog that does not exist yet.

Stack inconsistencies within the proposal itself

Issue Detail
Angular listed, React-Kendo omitted Proposal lists @luma/angular-kendo but not @luma/react-kendo, even though React+Kendo is the dominant Storybook stack.
Multi-framework without token strategy Unclear whether tokens stay framework-agnostic (CSS vars) while components are stack-specific, or whether each stack gets its own token format.
Web components not mentioned Changelog records “Removed Web Components export from build.” Draft arcadis-checklist Lit component still exists in src/web-components/ under “Draft Web Components” in Storybook.

  • Proposal: Separate @luma/fonts package — format and override rules undefined.
Angular icons compatibility gap

Stage 1 — Adoption Model

How product teams are expected to adopt Luma today vs. how the proposal describes adoption.

Current adoption path (5 steps)

1
Configure GitHub Packages auth (.npmrc + PAT + SSO)
2
Install @progress/kendo-theme-default
3
Install @arcadis-intelligence/luma-design-system
4
Import dist/luma-theme.css once
5
Use Kendo components directly in app code

Proposal assumes

  • Teams pick packages matching their stack
  • Versioned tokens, icons, fonts installed independently
  • Component and pattern packages reduce manual wiring
  • Incremental adoption across release cadences

Reality today

  • All-or-nothing CSS import (tokens not separable)
  • No migration path from current package to @luma/*
  • Page Shell / Wizard = copy from Storybook, not npm install
  • No codemods, no upgrade CLI, no compatibility matrix

Pattern & template deliverable ambiguity

Page Shell implementations (Top Bar, Page Header, Wizard, Collapse Sidebar) exist as:

  • Storybook stories with live React demos
  • template.ts files exporting code strings for copy-paste
  • Co-located SCSS not included in published dist/

The proposal says “template implementations” will be published packages, but does not define format: npm package? CLI scaffold? Storybook-only reference?

No defined migration path
Existing products using @arcadis-intelligence/luma-design-system have no documented path to @luma/tokens + stack-specific packages. Breaking changes, deprecation timeline, and aliasing strategy are all undefined.

Stage 2 — AI-Assisted Adoption

The proposal describes AI agents that analyze product codebases and generate migrations. What exists today is a Storybook FAQ chatbot with a static knowledge base.

Stage 2 capability (proposal) LumaAssistant today Status
Analyze product repositories No repo access; runs client-side in Storybook only Missing
Identify Luma adoption opportunities in real code Keyword-matched FAQ entries (~2,600 lines hardcoded KB) Missing
Recommend replacements for custom UI Answers “how to use Button” style questions from docs Missing
Assist with Luma version upgrades No version logic, migration spec, or codemods Missing
Generate implementation proposals and PRs No code generation or PR tooling Missing
Incremental migration toward Luma standards Not implemented Missing

Proposal framing

  • AI as engineering assistant (engineers own decisions)
  • Agents gain awareness of product repositories
  • Generate PRs and migration proposals
  • Accelerate alignment with Luma standards

Internal contradiction in proposal

  • Says “assistant, not automation” but also “generating pull requests”
  • No guardrails: repo access, SSO, secrets, compliance
  • No PR review workflow or validation requirements defined
  • No stack detection (Kendo vs Material vs shadcn per product)

Stage 2 depends on Stage 1 (circular dependency)

AI migration only works if agents know:

  • Stable package names and import paths per stack
  • Component mapping catalog (custom DataTable → Kendo Grid)
  • Token migration rules between versions
  • Codemods or machine-readable migration guides

None of these exist. Today there is one CSS import and no mapping catalog. Stage 2 cannot run meaningfully until Stage 1 defines stable package APIs.

“Luma equivalents” do not exist
Luma does not ship component equivalents — only styled Kendo primitives and layout recipes in Storybook. An AI agent would need a component mapping catalog that is not in the repository. Proposal language assumes equivalents exist; the codebase contradicts this.
Today → User asks in Storybook → Keyword match KB → Static doc answer
Proposed → Agent reads product repo → Maps to @luma/* packages → PR with migration

Recommended Next Steps

Minimum alignment work before engineering Stage 1 or Stage 2. Without these, the proposal and repository will remain out of sync.

1
Confirm stack strategy
Kendo-first vs multi-framework. If Kendo-first, add @luma/react-kendo to proposal and defer Material/shadcn.
2
Define package list
Exact @luma/* names, what each exports, and what stays in a legacy meta-package.
3
Define “component package”
Document whether teams get wrappers, CSS-only, or full libraries per stack.
4
Resolve naming
@luma vs @arcadis-intelligence and registry strategy.
5
Fix versioning
Align package.json, CHANGELOG, and release process before splitting packages.
6
Publish migration path
How existing CSS consumers move to @luma/* without breaking apps.
7
Scope Stage 2 separately
Treat as new platform (repo-aware agent + mapping catalog), not LumaAssistant v2.
8
Build component mapping catalog
Prerequisite for AI migration: custom patterns → Luma/Kendo equivalents.
Suggested phasing

Phase 0 (alignment): Answer decision checklist; fix version drift; update proposal to match Kendo reality.

Phase 1a (low risk): Extract @luma/tokens and @luma/icons from existing SCSS without breaking monolithic CSS.

Phase 1b: Monorepo + @luma/react-kendo / @luma/angular-kendo as documented adoption packages (theme + setup, not necessarily wrappers).

Phase 1c: Pattern packages or CLI for Page Shell templates.

Phase 2: Repo-aware AI with mapping catalog — only after package APIs are stable.
