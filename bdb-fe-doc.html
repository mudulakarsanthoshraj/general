5 — 🔌 All API Endpoints

All API calls are made through src/services/common/HttpCommon.ts — an Axios instance that automatically injects the Bearer token.
Prepend the base URL for your environment: Dev: https://devbdb.arcadis.com/api/bdbapi/api/  |  UAT: https://uatbdb.arcadis.com/api/bdbapi/api/  |  Prod: https://bdb.eu/api/bdbapi/api

⚠️ ID parameters — always get them from a list endpoint first
Every {id} in a URL is an internal database ID. You cannot guess it. Always call the list version first (e.g. GET /ReferenceProject) to get valid IDs, then call the detail endpoint.

What is BDB?

BDB (Bouw Database - Construction Database) is a comprehensive platform for the Dutch construction industry that provides:

  • Market Indicators: Real-time construction industry trends and analytics tracking labor costs, material prices, and economic factors affecting construction
  • Monthly Figures (Maandcijfers): Historical and forecast construction cost indices with labor/material breakdowns, showing trends over time to help predict future costs
  • Aspect Ratios (Kengetallen): Detailed cost breakdowns by construction categories (e.g., foundations, structure, finishing) across different project types and regions
  • Parametric Elements: Granular construction element cost analysis with searchable database of building components and their associated costs
  • Reference Projects: Project-based budget tracking allowing comparison against actual construction projects as benchmarks
  • Flexible Indices: Customizable index calculations where users can define their own weightings for labor/material/overhead to match their specific project profiles
  • WOV Index: Woningbouw, Onderhoud en Verduurzamingsindex - specialized index for housing construction, maintenance, and sustainability projects

Business Context

The BDB platform serves construction companies, cost consultants, project managers, and real estate developers in the Netherlands who need:

  • Accurate cost estimation: Base their project budgets on real market data rather than estimates
  • Trend analysis: Understand how construction costs evolve over time to plan long-term projects
  • Benchmarking: Compare their project costs against industry standards and similar projects
  • Budget forecasting: Predict future costs based on historical trends and forecast models
  • Risk management: Identify cost volatility in specific materials or labor categories
  • Competitive intelligence: Understand market positioning and pricing strategies

User Roles & Access

Standard User

  • Access based on company subscriptions
  • View and export data for subscribed products
  • Create and manage ad-hoc projects
  • Customize personal dashboard
  • Save custom presets and configurations
  • Limited to their company's data scope

Admin User

  • All standard user capabilities
  • Upload reference budget data
  • Upload forecast figures
  • Upload WOV index data
  • Bulk publish data across system
  • Access to all products regardless of subscription
  • Manage company-wide settings
Key Application Features
  • Multi-tenant architecture with company-based access control
  • Role-based permissions (Admin, Standard User)
  • Subscription-based feature access
  • Comprehensive data export capabilities (Excel, PDF)
  • Internationalized interface (Dutch/English support)
  • Customizable dashboard with drag-and-drop tiles
  • Session timeout management and security controls

3. Authentication & Authorization

Uses Microsoft Authentication Library (MSAL) for Azure B2C integration. Why Azure B2C?

  • Enterprise SSO: Users authenticate once across all Arcadis platforms
  • Security: OAuth 2.0 and OpenID Connect protocols with automatic token rotation
  • Token Management: MSAL automatically refreshes expired tokens without user interaction
  • Session Storage: Tokens stored in sessionStorage (cleared on browser close for security)
  • HTTP Interceptors: Axios interceptor automatically adds Bearer tokens to all API requests
  • Role-Based Access: Route guards (PrivateRoute) check authentication before rendering protected pages
Authentication Token Lifecycle

1. Login: User redirects to Azure B2C login page → enters credentials → receives JWT access token

2. Storage: Token stored in sessionStorage with expiry timestamp (typically 1 hour)

3. API Calls: Axios interceptor reads token from storage → adds Authorization: Bearer {'{token}'} header to every request

4. Refresh: MSAL silently acquires new token before expiry (happens in iframe without user seeing it)

5. Logout: Token cleared from storage → user redirected to Azure B2C logout endpoint

    redirectUri: window.location.origin,
    postLogoutRedirectUri: window.location.origin,

  cache: {
    cacheLocation: "sessionStorage",  // Token cleared when browser closes
    storeAuthStateInCookie: false,

4. State Management

The application uses a hybrid state management approach:

Global State (Redux Toolkit)

  • Purpose: Cross-cutting concerns only
  • Use Case: Snackbar/notification system
  • Why Redux here? Notifications can be triggered from any component, service, or error handler
  • Benefits: Single source of truth for notifications, easy to dispatch from anywhere
  • Store Setup: Minimal configuration, only snackbarSlice registered

Local State (Context API + Hooks)

  • Purpose: Feature-specific state
  • App Context: clientId (current project/client)
  • Component State: Form data, UI toggles, local filters
  • Custom Hooks: useMonthlyFigures, useMarketIndicator (data fetching + loading state)
  • Why not Redux? Avoids prop drilling, simpler than Redux for local concerns, better performance
State Management Decision Tree

Use Redux when: State needs to be accessed from multiple unrelated components across the app (e.g., notifications, global settings)

Use Context when: State is shared within a component tree (e.g., app-level settings like clientId)

Use Local State when: State is only needed in one component or its children (e.g., form inputs, modal open/close)

Use Custom Hooks when: Complex state logic needs to be reused (e.g., API data fetching with loading/error states)

Breadcrumb System

Custom breadcrumb support via BreadcrumbRouteObject type. How it works:

  1. Type Definition: Extends React Router's RouteObject with optional breadcrumb property

  3. Dynamic Breadcrumbs: Function that receives route params (e.g., breadcrumb: (params) => `Project ${params.id}`)
  4. Auto-Generation: Breadcrumbs component traverses route tree, collects all breadcrumb values from root to current page
  5. Navigation: Each breadcrumb is clickable → navigates to that route level

Services & API Integration

API service layer architecture and how the application communicates with the backend. All API calls go through service modules - components NEVER call axios directly.

Why the Service Layer Pattern?
  • Single Responsibility: Components focus on UI, services focus on data fetching
  • Testability: Easy to mock services in component tests (mock MonthlyFiguresService.getMonthlyFigures() → return fake data)
  • Reusability: Same service method used by multiple components (Dashboard tile + full page both call same service)
  • Maintainability: API endpoint changes only require updating service file, not 20 components
  • Error Handling: Centralized error handling, retry logic, timeout configuration
  • Type Safety: Services return typed data → TypeScript validates usage in components

Service Layer Architecture

All backend communication is abstracted into service modules located in src/services/. Each service handles a specific domain or feature area. All services extend the base HTTP client for consistent behavior.

HttpCommon.ts - Base HTTP Client Configuration

This is the foundation of all API communication. Every service imports this configured Axios instance.

import axios from 'axios';

// Create configured axios instance
const instance = axios.create({
  baseURL: import.meta.env.VITE_REACT_APP_API_URL,  // Dev: https://devbdb.arcadis.com/api/bdbapi/api/
  headers: {
    Accept: 'application/json',
    'Content-Type': 'application/json',
  },
  timeout: 30000,  // 30 second timeout for all requests
});

// REQUEST INTERCEPTOR - Runs before every API call
instance.interceptors.request.use(
  async (config) => {
    // 1. Check if token is expired, refresh if needed
    await authOperations.getRefreshToken();
    
    // 2. Get token from sessionStorage
    const token = appStorage.getAccessToken();
    
    // 3. Add Bearer token to Authorization header
    if (token) {
      config.headers.Authorization = `Bearer ${token}`;
    }
    
    // 4. Add clientId to URL query params (for multi-tenant API)
    const clientId = appStorage.getSelectedClientId();
    if (clientId) {
      config.params = { ...config.params, clientId };
    }
    
    return config;
  },
  (error) => Promise.reject(error)
);

// RESPONSE INTERCEPTOR - Runs after every API response
instance.interceptors.response.use(
  (response) => response,  // Success - pass through
  (error) => {
    // Handle common errors globally
    if (error.response?.status === 401) {
      // Unauthorized - token expired or invalid → redirect to login
      window.location.href = '/login';
    } else if (error.response?.status === 403) {
      // Forbidden - user lacks permission → show error snackbar
      dispatch(showSnackbar({ message: 'Access denied', severity: 'error' }));
    } else if (error.response?.status >= 500) {
      // Server error → show generic error message
      dispatch(showSnackbar({ message: 'Server error. Please try again.', severity: 'error' }));
    }
    return Promise.reject(error);
  }
);

export default instance;
Interceptor Benefits

Request Interceptor: Automatically adds auth token to EVERY API call → components don't need to handle authentication

Response Interceptor: Handles common errors (401, 403, 500) in ONE place → consistent error UX across entire app

Without Interceptors: Every component would need 20+ lines of auth + error handling code → massive duplication

Testing Strategy

Comprehensive testing approach using Vitest and React Testing Library. Testing Philosophy: Test behavior, not implementation.

Why This Testing Stack?
  • Vitest: Fast Jest-compatible test runner built for Vite → reuses Vite config → faster than Jest
  • React Testing Library: Tests components like users interact with them → queries by text, role, label (not by class/ID)
  • jsdom: Simulates browser DOM in Node.js → no need for real browser (fast, can run in CI)
  • Why not Enzyme? Enzyme tests implementation details (internal state, private methods) → brittle tests. RTL tests behavior (what user sees) → robust tests

  • Query by role/label/text: getByRole("button"), getByLabelText("Email"), getByText("Submit") - accessible to screen readers = robust tests
  • Avoid query by class/ID: getByClassName("btn-primary") - brittle, breaks when CSS changes
  • Fire events like users: fireEvent.click(button), fireEvent.change(input, {'{target: {value: "text"}}'})}
  • Wait for async: await waitFor(() => expect(...).toBeInTheDocument()) - for API calls, delays
  • Don't test implementation: Don't check component state, props structure - test what renders

🧪 Postman Guide — Testing BDB APIs from Scratch

A complete step-by-step guide for a developer with zero Postman experience. You will be making real BDB API calls within 10 minutes.

Step 0 — What is Postman?

Think of Postman as a browser for APIs

Instead of clicking through the app's UI, Postman lets you call backend API endpoints directly and see the raw JSON response. This is essential for:

  • Verifying what data the backend returns
  • Debugging API issues without running the frontend
  • Exploring what fields exist in a response
  • Testing admin-only endpoints

📥 Download: https://www.postman.com/downloads/ (free desktop app, no account needed to start)

Step 1 — Understand the Auth Challenge First

⚠️ Every single BDB API endpoint requires a Bearer token

BDB uses Azure B2C for login. There is no simple username/password API. The only practical way to get a valid token is to log in through the browser and copy the token from the browser's Network tab.

The token is a JWT string starting with eyJ... and it expires after ~1 hour. After expiry you need to grab a new one.

Step 2 — Get Your Bearer Token from the Browser

This is the most important step. Follow exactly:

🌐 Method A — Copy from Network Tab (Recommended, Easiest)
  1. Open https://devbdb.arcadis.com in Chrome or Edge
  2. Open DevTools → press F12 (or right-click → Inspect)
  3. Click the Network tab
  4. Log in with your Arcadis credentials
  5. After login, in the Network tab, look for any request going to devbdb.arcadis.com/api
    Hint: type api in the filter box at the top of the Network tab
  6. Click on any of those API requests (e.g. GetRoles)
  7. In the right panel → click Headers tab
  8. Scroll down to Request Headers → find Authorization
  9. The value looks like: Bearer eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9.eyJ...
  10. Copy everything after the word "Bearer " (including the space) — that's your token

💡 Tip: You can also open DevTools → Console tab and run:
Object.entries(sessionStorage).find(([k]) => k.includes('accesstoken'))?.[1]
then parse the returned JSON and copy the "secret" field value.

🔗 Understanding IDs — Why You Get 400 Errors

⚠️ The Most Common Mistake — Using the Wrong Kind of ID

Many BDB endpoints use {id} in the URL, but each endpoint expects a different type of ID. There is no universal ID — you must first call a list endpoint to discover valid IDs before calling a detail endpoint. Using a random number like 1 or 999 without checking will give you a 400 or 404.

Types of IDs used in BDB API

ID TypeWhat it isHow to get a valid oneEndpoints that use it
Reference Project ID The ID of a configured construction project used as a benchmark GET /ReferenceProject → pick any id from the response array ReportForecastFigures/{id}, ReferenceProject/getById/{id}, ReferenceDatePublished/getLatest/{id}
Reference Date ID The ID of a specific publication date/quarter (e.g. Q1-2024) GET /ReferenceDate → pick any id from the response array ReferenceDate/{id}
Data Series ID The version of the dataset (datasets are versioned per release) GET /DataSeries or GET /DataSeries/GetLastDefinitiveDataSeries → use the id field DataSeries/{id}
Upload Record ID (ReferenceBudget) The internal database row ID of a previously uploaded reference budget file. Does NOT exist until someone has uploaded files via the admin UI. POST /ReferenceBudget/take (see body below) → pick any id from response.list ReferenceBudget/getAll/{id}, ReferenceBudget/file/{id}, ReferenceBudget/delete/{id}
Upload Record ID (ForecastFigure) The internal row ID of an uploaded forecast figure file POST /ForcastFigure/take → pick any id from response list ForcastFigure/delete/{id}
Parametric Node ID The ID of a parametric element node in the tree GET /ParametricNode → pick any id from the response array ParametricNode/{id}, ParametricNode/{id}/CalculatePrice

🔍 Specific Fix: ReferenceBudget/getAll/{id} → 400 Error

Why you got 400 — and the exact 3-step fix

The problem: ReferenceBudget/getAll/{id} expects the internal upload record ID — a row ID that only exists if files have already been uploaded via the admin upload page. You cannot guess it or pick a random number.

Step 1 — Get the list of uploaded records first:

  • Method: POST
  • URL: {{baseUrl}}/ReferenceBudget/take
  • Body (raw JSON):
{
  "skip": 0,
  "take": 20,
  "sort": [],
  "filter": null
}

Step 2 — Look at the response:

{
  "list": [

Troubleshooting

Common issues and their solutions.

Development Server Issues

Port 3000 Already in Use

Solution

Kill the process using port 3000:

lsof -ti:3000 | xargs kill -9

Or change the port in vite.config.ts

Module Not Found Errors

Solution
  1. Delete node_modules/ and package-lock.json
  2. Run npm install again
  3. Restart the dev server

Authentication Issues

Infinite Login Loop

Possible Causes
  • Token expiration time not set correctly
  • Azure B2C configuration mismatch
  • Redirect URL not matching environment
  • SessionStorage being cleared unexpectedly

Solution:

  • Check VITE_REACT_APP_REDIRECT_URL matches current URL
  • Verify Azure B2C app registration settings
  • Clear browser cache and sessionStorage
  • Check console for MSAL errors

401 Unauthorized on API Calls

// Check if token is being sent
// Open browser DevTools → Network → Headers
# Look for: Authorization: Bearer [token]

// Debug token acquisition
console.log(appStorage.getAccessToken());
console.log(appStorage.getExpirationTime());

Build Issues

TypeScript Compilation Errors

Solution
  • Run npm run lint to see all type errors
  • Check tsconfig.json for strict mode settings
  • Ensure all imports have correct types
  • Use // @ts-expect-error only as last resort

Environment Variables Not Loading

Remember
  • Vite env vars must start with VITE_
  • Changes require server restart
  • Check you're using import.meta.env not process.env
  • Verify the correct .env.* file for your build command

Data Grid Issues (AG Grid)

License Warning in Console

AG Grid Enterprise requires a license. Check package.json for @mui/x-license package.

Grid Not Rendering

  • Ensure parent container has explicit height
  • Check data format matches column definitions
  • Verify AG Grid styles are imported

Translation Issues

Translations Not Loading

Checklist
  • Check network tab for i18n API calls
  • Verify solution ID and code in i18n.ts
  • Ensure translations exist on platform
  • Check browser console for i18next errors

Performance Issues

Slow Page Load

  • Check React DevTools Profiler for slow components
  • Look for unnecessary re-renders
  • Verify data fetching isn't happening in render
  • Use useMemo and useCallback appropriately

Large Bundle Size

  • Run npm run build and check dist/ folder size
  • Consider lazy loading heavy components
  • Check for duplicate dependencies
  • Use tree-shakeable imports (e.g., import { Button } from "@mui/material")
