The Ask

The Ask & Cost

Enable controlled access to one low-cost Large Language Model API (recommended: GPT-4o Mini or an approved alternative) so the team can explore and experiment with practical AI use cases. Today’s network restrictions (e.g. Zscaler / company policy blocking external API access) make even a small, safe experiment hard — this is a request to unblock thoughtful, hands-on learning.

Typical spend is ~$1–$30 per month under an agreed monthly budget cap (set at approval; indicative range $25–$50). The cap is a safety ceiling above expected usage, not a flat bill — see OpenAI Model Pricing for the per-token math. Leadership picks the final number.

What this unlocks

A small, controlled enablement helps both the organisation and individual growth by letting us:

  • Identify use cases for automation and productivity improvements
  • Explore how LLMs can be integrated into our existing applications
  • Support informed, cost-effective decisions on AI adoption
  • Encourage hands-on learning and innovation within the team

Done in a controlled manner

  • Minimal and monitored usage — capped spend with an audit trail
  • No sensitive or company data is shared with the model
  • Full compliance with company security policies

The same controlled key can later support more than one experiment — each vetted use case rides the same key, the same cap, and the same guardrails, so the marginal cost of the next one is near zero. Whether any of them pay off is measured per use case (see How We’ll Measure Value); candidate use cases in our existing products are in Real-Time Use Cases.

Scope of this Ask

What we’re asking management to approve — and what we are explicitly not asking for. Network policy (Zscaler) and existing security controls still apply.

To avoid any scope confusion: this is a tightly bounded enablement — one approved outbound API endpoint, used server-side for controlled exploration and experimentation. It is not a request to lift network policy, and it is separate from any conversation about personal AI tooling (Copilot, Claude, ChatGPT).

What this proposal asks for

  • Outbound access to one approved API endpoint — server-side; minimal, monitored usage for team learning and use-case discovery.
  • A Zscaler exception via the standard Security exception process for that one URL.
  • A reusable pattern so future vetted use cases can share the same enablement (operational guardrails in How It Stays Safe).

What this proposal does not ask for

  • Per-engineer or per-laptop API keys.
  • GitHub Copilot, Claude desktop, or ChatGPT in the browser — separate decisions with different trade-offs.
  • A blanket lift of Zscaler or any company-wide network policy.
  • Access to multiple vendors at once — one vendor, one URL, reviewed.
On Zscaler / network policy
We are not bypassing Zscaler. We use the same exception process as other approved SaaS APIs to allow one outbound URL for one reviewed use case. Network policy continues to apply everywhere else. Personal AI tooling remains a separate conversation — this proposal does not pre-decide it.

Why GPT-4o mini
For grounded Q&A, summarization, and extraction, GPT-4o mini delivers flagship-class answer quality at ~17× lower cost than GPT-4o and ~100× lower than the reasoning models. Even cheaper nano-class models exist, but answer quality drops on multi-clause requests that combine several constraints. GPT-4o mini hits the sweet spot — strong answers, predictable spend, comfortably inside a small agreed monthly cap.

Ollama — Free Local Alternative

Ollama is a free, open-source way to run Large Language Models on your own machine. It is excellent for development — this section explains why it can’t be the production answer for a real-time product feature, and how the two fit together.

ollama.com runs open models (Llama 3, Mistral, Phi, etc.) on a developer’s machine — no API key, no per-request cost, data never leaves the laptop. It supports OpenAI-compatible APIs, so the same application code can target Ollama locally and a hosted API in production.

Why Ollama can’t be the production deployment

What Ollama is great for

  • Prompt iteration and offline exploration on your own hardware
  • Useful when outbound API access is blocked — entirely local, no Zscaler exception needed

Why it falls short for production

  • A product feature can’t depend on each end-user installing Ollama and downloading a ~2–8 GB model — impractical
  • Performance depends on each machine (CPU/GPU/RAM); responses can take 10–30 seconds on a typical laptop — too slow for real-time UX
  • Smaller open models give noticeably lower answer quality than GPT-4o Mini for the same prompt
  • Hosting Ollama centrally on a shared server brings back ops + GPU costs that far exceed a small hosted API budget
  • No vendor SLA, no audit tooling, no managed updates
How both fit together
Use Ollama free for local development and prompt iteration. Use the low-cost GPT-4o Mini API for the production feature. Because both speak the OpenAI-compatible API, switching providers is essentially a one-line URL change, so we keep the freedom to revisit later as open models improve.

How AI Works in Real Time

“Real-time” AI means a user does something in the product and gets a useful, generated response within a second or two — an answer, a summary, a draft, extracted fields. Under the hood it is a short, predictable request loop to one Large Language Model API.

The request loop

Every real-time AI feature, regardless of product, follows the same five steps on the server side:

  1. Capture input — the user types a question, opens a long document, pastes text, or triggers an action.
  2. Retrieve context (grounding) — the app fetches the few most relevant pieces of our own data/docs for that input (Retrieval-Augmented Generation). This is what keeps answers accurate and on-topic.
  3. Build the prompt — a fixed system instruction + the retrieved context + the user input are combined into one request.
  4. Call the model — the request goes to the approved API (server-side, with the shared key and budget cap).
  5. Stream the response — tokens come back and render as they arrive, so the user sees the answer forming immediately rather than waiting for the whole thing.
Why it feels instant
Two things create the “real-time” feel: streaming (text appears word-by-word in <1s) and grounding (only a small, relevant slice of data is sent, so the model responds fast and stays accurate). A typical call is ~2,000 input + ~600 output tokens — a fraction of a cent on GPT-4o Mini.

What stays server-side

The API key, the prompt assembly, the retrieval, and the budget cap all live on the server — never in the browser or on a user’s machine. The product UI only sends the user input and renders the streamed answer. This is what makes a single approved endpoint safe to share across multiple product features.

How We’ll Measure Value

Every experiment is measured the same way: a baseline first, weekly tracking during the pilot, then an explicit go / no-go at week 4 — so learning is documented and we only scale what proves useful.

The four phases

Phase When What we measure
1. Baseline Week 0 — before pilot starts The current state for the chosen use case (e.g. time/clicks to complete the task, manual effort, volume) so there is a number to compare against.
2. Pilot tracking Weeks 1 – 4 Feature usage; weekly accuracy spot-check on a sample of 20 responses; API spend; the use case’s primary value metric.
3. Decision gate End of week 4 Explicit go / no-go against the thresholds below. If thresholds aren’t met, the feature is turned off and the API key is revoked.
4. Scale tracking Weeks 5 – 8 Same metrics across the full user group; confirm the pattern scales; final value and cost reported to management at week 8.

Decision gate — end of week 4

Concrete, written-down thresholds. We continue only if all three are met:

Value metric
Beats baseline
Answer accuracy
≥ 70%
API spend
≤ agreed cap

“Value metric” = the use case’s primary measure (time saved, faster completion, deflected manual work) vs the week-0 baseline. “Accuracy” = % of sampled responses the owning team rates as correct and grounded in our data (random 20-response sample, weekly).

If the numbers don’t hold up
The feature is switched off at the end of week 4, the API key is revoked, and the use case is closed. Sunk cost is capped at the agreed monthly ceiling over the pilot — no infrastructure, no licensing, no headcount commitments. This decision gate is in writing so it’s real, not aspirational.

What gets reported to management

  • Weekly during pilot: one-line status — usage, accuracy sample, value metric, spend-to-date.
  • End of week 4: written go / no-go memo against the three thresholds above.
  • End of week 8: final report — baseline vs measured value, actual API spend, and recommendation on next use cases.

Security & Cost Controls

Eight controls that bound the spend, the data, and the answer quality.

Control How it works
Hard monthly budget cap $50 ceiling on the API key; automatic shut-off when reached.
Data sent to the model Only the approved data for that use case and the user’s input. No customer data or product source unless explicitly classified and approved.
No training on our prompts Vendor’s “zero data retention” / no-training tier.
Approved vendor only Single vendor, terms reviewed by Security & Legal.
Audit trail All prompts and responses logged 30 days for security review.
Personal data filter (names, emails, employee IDs, etc.) Prompts scanned and blocked when they match known sensitive patterns.
Answers grounded in approved data only Retrieval-Augmented Generation — the model is fed relevant snippets from the approved source before each answer, so it can’t invent facts.
Unknown questions & quality review Clear “I don’t know” fallback when no source matches; weekly response sampling by the owning team; user-reportable answers.
