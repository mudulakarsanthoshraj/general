The Ask & Cost

We want to turn on safe access to one low-cost AI model (we suggest GPT-4o Mini, or a similar approved one) so the team can try out real AI ideas. Right now our network rules (like Zscaler blocking outside connections) make even a small, safe test hard. This is a request to open the door for careful, hands-on learning.

We expect to spend about $1–$30 a month, with an agreed monthly limit set when this is approved (somewhere around $25–$50). The limit is a safety ceiling, not a fixed bill — see OpenAI Model Pricing for the numbers. Leadership decides the final figure.

What this lets us do

This small, controlled access helps both the company and the team by letting us:

  • Find tasks we can automate or speed up
  • See how AI can fit into the apps we already have
  • Make smart, low-cost choices about using AI
  • Give the team a chance to learn by doing

Kept under control

  • Small, watched usage — capped spend with a full record
  • No private or company data is sent to the model
  • Follows all company security rules

The same access key can later cover more than one test. Each approved idea uses the same key, the same limit, and the same safety rules, so adding the next one costs almost nothing. We check whether each one is worth it (see How We’ll Measure Value); ideas for our existing products are in Real-Time Use Cases.

Scope of this Ask

What we are asking management to approve — and what we are not asking for. Our network rules (Zscaler) and security controls still apply.

To keep this clear: we are asking for one small, well-defined thing — access to one approved AI service, used only on our server, for careful testing and learning. We are not asking to change network rules, and this is separate from any talk about personal AI tools (Copilot, Claude, ChatGPT).

What we are asking for

  • Access to one approved AI service — used on our server; small, watched usage for team learning and finding good ideas.
  • A Zscaler exception for that one web address, using the normal Security approval steps.
  • A reusable setup so future approved ideas can share the same access (safety rules in How It Stays Safe).

What we are not asking for

  • A separate key for every engineer or laptop.
  • GitHub Copilot, Claude desktop, or ChatGPT in the browser — those are separate decisions.
  • Removing Zscaler or any company-wide network rule.
  • Access to many vendors at once — just one vendor, one address, reviewed.
About Zscaler / network rules
We are not getting around Zscaler. We use the same approval steps as other approved cloud services to allow one web address for one reviewed use case. All other network rules stay in place. Personal AI tools are a separate topic — this request does not decide them.

Why GPT-4o mini
For answering questions, summarizing, and pulling out data, GPT-4o mini gives top-quality answers at about 17× lower cost than GPT-4o and around 100× lower than the reasoning models. There are even cheaper models, but their answers slip when a request has several parts. GPT-4o mini is the sweet spot — strong answers, steady cost, and easily within a small monthly limit.
What a low-cost model like GPT-4o mini is good for
Low-cost models are great for things like AI agents and chatbots, finding and filtering data, summarizing, and pulling out details — but they are not built to write complex code. For heavy coding or hard problem-solving, a bigger model (GPT-4.1 or a reasoning model) is the better choice. Match the model to the job: use GPT-4o mini for everyday assistant and data tasks, and save the pricier models for the few cases that really need them.

Ollama — Free Local Alternative

Ollama is a free, open-source tool for running AI models on your own computer. It is great for building and testing — this section explains why it can’t run the live product feature, and how the two work together.

ollama.com runs open models (Llama 3, Mistral, Phi, and others) right on a developer’s computer — no key, no per-use cost, and the data never leaves the laptop. It works the same way as the OpenAI API, so the same code can use Ollama while building and a hosted model when live.

Why Ollama can’t run the live feature

What Ollama is great for

  • Trying out prompts and ideas offline on your own computer
  • Handy when outside access is blocked — it’s all local, so no Zscaler exception is needed

Why it falls short for the live product

  • We can’t expect every user to install Ollama and download a 2–8 GB model — that’s not practical
  • Speed depends on each person’s computer; answers can take 10–30 seconds on a normal laptop — too slow
  • Smaller open models give clearly weaker answers than GPT-4o Mini for the same question
  • Running Ollama on a shared server brings back server and GPU costs that are far higher than a small hosted-model budget
  • No vendor support, no audit tools, no managed updates
How both work together
Use Ollama (free) while building and testing, and the low-cost GPT-4o Mini for the live feature. Since both work the same way, switching between them is basically a one-line change — so we can revisit this later as open models get better.

How It Stays Safe

Safety rules built into every use case: one approved model, a spending limit, no private data, and a sign-off before anything goes live.

Safety rule How it works
One approved model We start with GPT-4o Mini only. Any new model goes through Security review.

No private data Only public docs and made-up sample data — no personal details (names, emails, employee IDs), no private code, no passwords.
Sign-off for each use case A short one-page plan (owner, what data is used, spending limit, success measure) is approved by Security, Platform, and the product owner before going live.

How AI Works in Real Time

“Real-time” AI means the user does something in the app and gets a useful reply in a second or two — an answer, a summary, a draft, or some extracted details. Behind the scenes it’s a short, repeatable set of steps that calls one AI model.

The steps

Every real-time AI feature, in any app, follows the same five steps on our server:

  1. Take the input — the user types a question, opens a long document, pastes text, or clicks a button.
  2. Pull in the right context — the app grabs the few most relevant pieces of our own data or docs for that input. This is what keeps answers accurate and on-topic.
  3. Put the request together — a fixed instruction + that context + the user’s input are combined into one request.
  4. Send it to the model — the request goes to the approved model (on our server, using the shared key and spending limit).
  5. Show the answer as it comes — the reply appears word by word, so the user sees it forming right away instead of waiting for the whole thing.
Why it feels instant
Two things make it feel instant: the answer appears word by word (in under a second) and we only send a small, relevant slice of data, so the model replies fast and stays accurate. A typical call is about 2,000 input + 600 output tokens — a fraction of a cent on GPT-4o Mini.

What stays on our server

The key, the request building, the data lookup, and the spending limit all stay on our server — never in the browser or on a user’s computer. The app only sends the user’s input and shows the answer. That’s what makes one approved connection safe to share across several features.

Every flow uses the same five steps — take the input, pull in our own context, send it to the approved model on our server, and return the result. Only the lookup and output steps change for each pattern; the key, limit, and safety rules stay the same.

Important — what data is used Every use case must say what data it sends to the model. We start with public or low-risk data only. Anything with customer or personal data goes through Security review and the personal-data filter before it can go live (see Security & Cost Controls).

All of these share one key, one limit, one set of safety rules. The first approved test proves the idea; each one after that is cheaper to add because the setup is already there.

Example — using GPT-4o mini in a product

A clear, useful example is a plain-language booking flow: the user describes what they want in everyday words, GPT-4o mini turns it into neat, organized data, the user confirms, and only then is it saved. No forms to fill — the user just types a sentence.

1
User describes the trip in everyday words — the app sends the sentence to our server.
2
GPT-4o mini turns it into data — one call on our server returns the trip as neat, organized fields.
3
User confirms — the details are shown back; the user checks them and clicks Confirm.
4
Saved — only after the user confirms is the booking request stored.

Steps 1–2: turn the user's words into data (JSON)

response_format: json_object makes the model return clean, organized data. The key stays on our server, never in the browser. 

// userText = "I'd like to fly to Canada on 15 June. Please find the
//            cheapest flight, include meals, and any departure time works for me."
const res = await fetch("https://api.openai.com/v1/chat/completions", {
  method: "POST",
  headers: {
    "Authorization": `Bearer ${process.env.OPENAI_API_KEY}`,
    "Content-Type": "application/json"
  },
  body: JSON.stringify({
    model: "gpt-4o-mini",
    temperature: 0,
    response_format: { type: "json_object" },
    messages: [
      { role: "system",
        content: "Convert the travel request to JSON with keys: destination, travel_date (YYYY-MM-DD), transport_mode, budget_preference, include_meals (boolean), time_preference. Use null if not stated." },
      { role: "user", content: userText }
    ]
  })
});
const booking = JSON.parse((await res.json()).choices[0].message.content);

Steps 3–4: confirm, then save

The details are shown back to the user. Nothing is saved until they confirm — the AI drafts, the person decides. 

// Only runs after the user reviews and clicks "Confirm"
if (userConfirmed) {
  await db.bookings.insert({
    destination:       booking.destination,
    travelDate:        booking.travel_date,
    transportMode:     booking.transport_mode,
    budgetPreference:  booking.budget_preference,
    includeMeals:      booking.include_meals,
    timePreference:    booking.time_preference,
    createdBy:         currentUser.id,
    status:            "pending"
  });
}
Why this works well — and stays safe
  • Easy for users: they type one sentence instead of filling a long form — faster, with fewer people giving up.
  • Cheap: a request like this is about 300 input + 80 output tokens, well under a tenth of a cent on GPT-4o mini (see OpenAI Model Pricing).
  • Under control: the model only reads and organizes the text — it never saves anything. Data is saved only after the user confirms, on our server, under one limited key.

How It Helps

Clear wins for the team and the company once safe AI access is available.

  • Real learning. Engineers try real AI ideas instead of just reading about them or hitting a blocked connection.
  • Better decisions. We measure cost, quality, and safety from real use before making bigger commitments.
  • Faster results. One setup (key, limit, safety rules) is reused instead of each team sorting out access on its own.
  • Finding good ideas. We spot ways to automate and save time by working hands-on in the apps we already have.
  • Measured and reversible. Every test has a limit and a success measure; we keep what works and switch off what doesn’t.
  • A path to AI in our products. Once the idea works on a low-risk test (like a read-only helper), the same approach can grow to bigger uses.

How We’ll Measure Value

We measure every test the same way: take a starting number first, track it weekly during the test, then make a clear keep-or-stop call at week 4 — so we write down what we learn and only grow what actually helps.

The four steps

Step When What we measure
1. Starting point Week 0 — before the test starts How things work today for the chosen task (like time or clicks to finish it, manual effort, how often it happens) so we have a number to compare against.
2. Track the test Weeks 1 – 4 How much it’s used; a weekly quality check on a sample of 20 answers; the spend; and the main measure of value for the task.
3. Keep-or-stop call End of week 4 A clear keep or stop decision against the targets below. If the targets aren’t met, we turn the feature off and cancel the key.
4. Track at scale Weeks 5 – 8 Same measures across all users; check it still works at scale; final value and cost reported to management at week 8.

Keep-or-stop call — end of week 4

Clear targets, written down ahead of time. We continue only if all three are met:

Value
Beats the start
Answer quality
≥ 70%
Spend
≤ agreed limit

“Value” = the main measure for the task (time saved, faster completion, less manual work) compared with the week-0 starting number. “Quality” = the share of sampled answers the team rates as correct and based on our data (a random sample of 20 answers, checked weekly).

If the numbers don’t hold up
We turn the feature off at the end of week 4, cancel the key, and close the use case. The most we can lose is the agreed monthly limit over the test — no servers, no licenses, no new hires. This keep-or-stop call is written down, so it’s real and not just a hope.

What we report to management

  • Weekly during the test: a one-line update — usage, quality sample, value, and spend so far.
  • End of week 4: a short written keep-or-stop note against the three targets above.
  • End of week 8: a final report — starting point vs measured value, actual spend, and a suggestion on what to do next.

Security & Cost Controls

Eight simple rules that keep the spend, the data, and the answer quality under control.

Control How it works
Monthly spending limit The API key has a fixed monthly cap. When it’s reached, it stops on its own.
What we send to the model Only the approved data for that task and what the user types. No customer data or product code unless it has been checked and approved first.
Our data is not used for training We use the vendor’s “no-training” tier, so our prompts are not kept or used to train their models.
One approved vendor only We use a single vendor whose terms have been reviewed by Security and Legal.
Full record of activity Every prompt and response is logged for 30 days so Security can review it.
Personal data is blocked Prompts are scanned for personal details (names, emails, employee IDs, etc.) and blocked before they are sent.
Answers come only from approved data The model is given the relevant approved information before it answers, so it can’t make facts up.
Honest “I don’t know” and regular checks If there’s no matching source, it says “I don’t know” instead of guessing. The team reviews a sample of answers each week, and users can flag bad ones.
